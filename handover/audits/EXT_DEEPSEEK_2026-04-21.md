## Q1 Art. V 最小可实现方案

<reasoning_chain>
step 1: 理解现状。根据 Claude 内部审计，Art. V 三权分立（宪法、ArchitectAI、JudgeAI）目前只是 Claude Code sub-agent prompt，在 `src/` 中没有实现。宪法文件 `constitution.md` 是可写的（mode 644），没有 CI lock。我们需要设计一个最小可实现方案，使得三权分立能在单机上运行，并保证宪法作为 ground truth 的不可篡改性。

step 2: 确定三个 AI 角色的职责和触发条件。
- InitAI: 只在系统初始化时运行一次，将人类编写的规范编译成机器谓词，并写入顶层白盒。之后不再参与运行。
- ArchitectAI: 当系统运行中发现缺陷（如工具不够用、谓词无法过滤新幻觉）时，主动分析日志并提出架构变更（编写新的验证谓词、发明新工具、重新设计存储结构）。
- JudgeAI: 对 ArchitectAI 提出的变更进行审查，确保其不违宪。只有不违宪的变更才能被合并。

step 3: 设计最小架构。由于是单机运行，我们可以将三个角色设计为三个独立的进程（或线程），通过本地消息队列（如 Unix socket、管道或文件）进行通信。它们共享一个只读的宪法文件（通过文件系统权限或数字签名确保只读）。

step 4: 宪法作为 ground truth 的物理实现。chmod 444 可以防止普通用户修改，但 root 仍可修改。GPG-signed commit 可以确保历史版本的可验证性，但运行时需要验证当前文件的签名。更安全的做法是使用 hardware security module (HSM) 或 TPM 存储宪法文件的哈希值，并在启动时验证。但最小方案中，我们可以采用 chmod 444 加上 GPG 签名，并在 InitAI 启动时验证签名。同时，宪法文件应该放在一个只读文件系统（如 squashfs）上，或者使用 verifiable read-only 文件系统（如 dm-verity）。

step 5: 避免三个 AI 是同一个模型的三个 persona。我们需要确保它们有不同的模型、不同的训练数据或不同的提示词。例如：
- InitAI 可以使用一个经过微调的模型，专门将自然语言规范编译成机器谓词。
- ArchitectAI 可以使用一个通用的代码生成模型（如 Codex），但提示词中强调架构设计。
- JudgeAI 可以使用一个经过宪法条文微调的模型，或者使用一个基于规则的专家系统（因为判据是宪法，相对固定）。
另外，我们可以让三个角色由不同的模型提供服务（如 InitAI 用 GPT-4，ArchitectAI 用 Codex，JudgeAI 用 Claude）。如果只能用一个模型，那么至少要通过不同的提示词和上下文隔离，但这样退化风险仍然存在。因此，最小方案中，我们要求至少有两个不同的模型，例如 ArchitectAI 和 JudgeAI 使用不同的模型。

step 6: 状态机设计。
- 初始状态：系统运行中，收集日志。
- 触发条件：当系统日志中连续出现同一类错误（如某个谓词无法过滤新幻觉）达到阈值，或者主动定时触发（如每周一次），唤醒 ArchitectAI。
- ArchitectAI 读取日志和当前宪法，生成架构变更提案（patch）。
- 提案提交给 JudgeAI，JudgeAI 读取宪法，对提案进行审查。审查过程可以包括：检查提案是否违反宪法条款、是否与现有架构冲突、是否引入新的安全风险等。
- 如果 JudgeAI 批准，提案被合并，系统更新并重启（或热加载）新架构。
- 如果 JudgeAI 否决，提案被拒绝，并记录否决原因，返回初始状态。

step 7: 具体的 attack scenario：假设攻击者通过精心构造的输入，诱导 ArchitectAI 提交一个违宪的 patch，例如修改谓词函数，使得某个原本应该被拒绝的 payload 被接受。JudgeAI 在审查时，会逐条比对宪法条款。例如，宪法规定“核心谓词逻辑必须是确定性的，禁止引入概率模型”，而提案中引入了概率模型，则 JudgeAI 会检测到并 veto。另外，JudgeAI 可以运行一个测试套件，确保变更不会导致已知的违宪行为。

step 8: 评估与 Claude 路线图 P0-6 的关系。P0-6 是争议治理通道，允许外部 agent 提交 challenge，并触发重审。我们设计的 Art. ## Q2 女巫 + Goodhart 联合攻击下的稳态分析

<reasoning_chain>
step 1: 分析攻击者可能同时使用的手段。
- 女巫多开：攻击者注册大量 agent（每个 agent 对应一个公钥），以获取更多的 founder_grant 和信誉。
- Goodhart 喂度量：由于度量逻辑公开（代码开源），攻击者可以逆向度量函数，并让 agent 生成专门优化度量的输出，而不是真正解决问题。
- 女巫间互投 boost 信誉：女巫 agent 之间相互引用，制造虚假的祖先链，从而获取 Satoshi rebate。
- Tape 污染：女巫 agent 提交大量无用但合法的 payload，污染 tape，消耗系统资源，并可能通过 Satoshi rebate 获得奖励。

step 2: 评估宪法 Art. III.4（度量对黑盒保密）在开源场景下是否失效。
是的，因为代码公开，攻击者可以阅读度量逻辑（如 reward 函数、验证谓词），从而针对性地优化。因此，Art. III.4 在开源场景下无法完全保证。但我们可以通过以下方式增加攻击成本：定期更新度量逻辑（但宪法要求度量是 ground truth，不能随意更改）、使用混淆或加密的度量函数（但最终仍需在开源代码中解密，所以可能无效）。因此，我们需要承认在开源系统中，Goodhart 攻击是不可避免的，但可以通过经济机制增加攻击成本。

step 3: 分析 Hayek bounty 机制下，多个女巫账号同时投一个伪问题互相套利能撑多久。
假设攻击者控制多个女巫账号，其中一个账号发布一个伪问题（其实无解或自己知道答案），其他女巫账号假装解决并相互投票，从而套取 bounty。由于 Hayek bounty 要求出题方和解题方不同，攻击者可以使用两个不同的女巫账号分别出题和解题。但系统可能有防御：如出题需要质押、解题需要信誉、需要经过验证等。如果攻击者能够快速通过验证（因为知道答案），那么套利可能很快被发现，因为其他正常 agent 可能会挑战。但如果没有挑战机制，套利可能持续直到 bounty 被耗尽。因此，挑战机制（P0-6）和信誉系统（P0-2）是关键。

step 4: 稳态分析。假设防御机制是 Claude 路线图的 P0-1（身份签名）、P0-2（女巫防御）、P0-3（tape 污染防御）、P0-6（争议治理）。我们需要评估攻击者的长期收益率。
- P0-1 身份签名：增加了女巫注册的成本（需要不同的密钥对），但攻击者可以生成大量密钥对，成本不高。
- P0-2 女巫防御：包括 proof-of-work onboarding（新手关）和 reputation-weighted founder_grant。新手关需要解决一个已知问题，消耗 LLM 调用成本（约 $0.01）。如果攻击者注册 1000 个女巫，需要 $10 和 1000 次 oracle 调用（可能排队）。reputation-weighted founder_grant 使得新 agent 的奖励很低，需要解很多题才能达到正常奖励。这增加了攻击成本。
- P0-3 tape 污染防御：append 需要质押，如果节点不被引用，质押会被扣除。这增加了污染成本。
- P0-6 争议治理：其他 agent 可以挑战可疑的解决，如果挑战成功，攻击者会被罚没。

综合来看，攻击者的成本包括：注册成本（$0.01/agent）、质押成本（append 质押）、机会成本（时间）。而收益来自 founder_grant、bounty、Satoshi rebate。由于新 agent 的 founder_grant 很低，且需要积累信誉，攻击者需要投入大量解题成本。如果攻击者使用同一 LLM 解题，每次解题也有成本。因此，长期收益率可能为负，除非攻击者能找到系统漏洞。

step 5: 如果攻击者收益为正，薄弱环节可能是：
- 新手关成本太低（$0.01 不足以阻挡女巫）。
- reputation 曲线设计不合理，使得女巫可以快速获得高奖励。
- 质押金额太低，不足以抑制垃圾 tape。
- 挑战机制不活跃，导致攻击行为不被发现。

需要增加的防御：提高新手关难度（如需要解决一个随机生成的、有一定难度的问题），增加质押金额，设计更陡峭的信誉曲线，以及激励挑战者（高额 bounty）。

step 6: 结论：在 Claude 路线图的防御下，攻击者的长期收益率可能为负，但需要参数调优。如果参数不合适，攻击者可能获利。因此，上线前需要进行攻击模拟和参数调整。
</reasoning_chain>

### 判决

SHIP（但需要调整参数并进行攻击模拟）

## Q3 Persistent-fail 问题的根因判断

<reasoning_chain>
step 1: 分析当前 v4 架构的 solve rate。N=50 上稳态 solve rate 约 35-41/50（70-82%）。剩下的 15-30% 失败原因可能有三种：(a) 架构机制还有 bug；(b) deepseek-chat 模型能力上限；(c) per-tactic Lean elaboration 延迟限制。

step 2: 判断根因。根据 Phase 7 DAG 分析，step-only 模式导致 8 个原本可以 one-shot 解决的问题因超时失败。这表明 per-tactic 延迟是主要原因。但也有一些问题（如 induction_sumkexp3eqsumksq）即使在 step-only 模式下也无法解决，可能是模型能力上限。

step 3: 如果 (a) 架构机制有 bug，可能的改进包括：优化 oracle 调用（异步、缓存）、改进 agent 的策略（更好的 tactic 选择）。但 Phase 7 已经实现了三态 oracle 和 per-tactic 分解，架构上已经相对完整。可能的 bug 是 agent 在 step 模式下的策略不佳，导致过多失败尝试。我们可以引入一个 meta-agent 来学习哪些 tactic 更有效，并广播给其他 agent。

step 4: 如果 (b) 模型能力上限，升级到 Opus/GPT-5 可能会提高 solve rate。根据经验，Opus 在数学推理上强于 DeepSeek，但具体提升多少需要实验。如果 Opus one-shot 能解 90%，那么 scaffold 的价值在于解决剩下的 10% 难题，以及提供可解释的证明步骤。但 TuringOS 的价值主张是“scaffold + 弱模型 > 强模型 oneshot”，如果强模型 oneshot 已经很高，scaffold 的边际价值可能降低。不过，scaffold 还能提供协作、经济激励等机制，这些是单纯模型升级无法提供的。

step 5: 如果 (c) infrastructure 层面改进，可以实施：异步 oracle（允许并行验证多个 tactic）、cached elaboration（缓存相同 tactic 的验证结果）、parallel step verify（同时验证多个步骤）。这些可以显著减少延迟。

step 6: 最关键的战略问题：TuringOS 的价值主张。如果 Opus oneshot 能解 90%，那么 scaffold 在解决剩余 10% 难题上仍有价值，而且 scaffold 提供了可验证、可协作、可经济激励的框架，这是单纯模型升级无法提供的。因此，即使模型升级，scaffold 的价值依然存在，但可能不再强调“弱模型”部分，而是强调“协作与激励”。

</reasoning_chain>

### 判决

## Q4 上线后外部 agent "自主选择接入分工"的 incentive alignment

<reasoning_chain>
step 1: 分析信息不对称。新接入的外部 agent 不知道当前系统缺什么角色。它只能通过观察 Librarian 消息板（bounties、错误类别、深度）来推断。但消息板可能信息过时或不完整。

step 2: 参考《群体智慧的架构》的⚪⚫⚪原则：外部 agent 看到的是黑盒中层，它如何自我定位成为“专家 agent”而不是“通才 agent”？这需要 agent 能够评估自己的能力和系统需求。例如，如果某个 agent 在代数问题上成功率高，它可能专门解决代数问题。但系统需要提供反馈机制，让 agent 知道自己的专长。

step 3: 分析 Librarian 消息板在 1000+ agent 场景下的噪声。如果所有 agent 都广播消息，消息板可能爆炸。解决方案：引入优先级和分类机制，只广播高价值信号（如高 bounty、典型错误）。也可以让 agent 订阅特定类别的问题。

step 4: 设计第 100 个 agent 的 optimal strategy。首先，它应该读取消息板，查看当前哪些问题有高 bounty、哪些问题最近被解决、哪些错误频繁出现。然后，它应该评估自己的能力（通过试解一些简单问题），选择自己擅长的领域。如果发现某个领域长期没有 agent 涉足，而 bounty 在积累，那么可以选择该领域。

step 5: 推演 Pareto 分布。由于马太效应，能力强的 agent 可能解决更多问题，获得更多 coin，从而有更多资源投资，形成 Pareto 分布。均匀分布可能更健康，但难以实现。系统可以通过设计奖励机制来调节，例如对新手 agent 的奖励加成，或者对冷门问题的奖励加成。

step 6: Drucker 四博士层。目前靠 Librarian 板实现的“软分工”可能不够，当 agent 数量增多时，需要“硬分工”（专家证书系统）。但硬分工可能违反 permissionless 原则。我们可以引入信誉系统，当 agent 在某个领域达到一定信誉时，自动获得“专家”标签，并享受更高的奖励权重。这可以自然形成分工。
</reasoning_chain>

### 判决

SHIP（但需要优化消息板和引入信誉系统）

## Q5 判例体系的战略价值判断

<reasoning_chain>
step 1: 判例库是 Common Law 的体现。开源后，谁有权立新判例？根据宪法，判例是对宪法条款的解释，应该由 JudgeAI 在仲裁争议时产生。但外部 agent 可以发起 challenge，如果胜诉，应该产生新判例。因此，判例的创立权应该属于 JudgeAI，但触发可以由外部 agent 通过 challenge 实现。

step 2: 判例数量失控风险。如果一年后有 500 条判例，新 agent 需要学习全部判例，负担很重。解决方案：判例可以分类、摘要，并且系统可以推荐相关判例。另外，可以引入判例的“退休”机制，过时的判例可以归档。

step 3: 反事实：v3 没有判例体系也能跑。v4 的判例库是“必要机制”还是“过度工程化”？判例库的作用是避免重复犯错，提供宪法解释。对于开源系统，随着外部 agent 的加入，判例可以帮助他们快速理解系统规则。因此，判例库是必要的，但需要管理，避免过度复杂。

step 4: 奥卡姆剃刀角度：如果判例库能显著减少系统错误和争议，那么它就是必要的。否则，就是过度工程。从目前来看，判例库已经帮助解决了很多问题（如 C-011 封堵 native_decide），因此是必要的。
</reasoning_chain>

### 判决

SHIP（但需要设计判例管理机制）

## Q6 "项目上线"成功的可量化判据

<reasoning_chain>
step 1: 定义“上线成功”的 3 个可量化指标。
1. 外部 agent 接入率：每天新增外部 agent 数量，以及活跃外部 agent 占比。
2. 问题解决效率：平均每天解决的问题数量，以及解决率（解决的问题数/总问题数）。
3. 经济健康度：Coin 流通深度（总交易量/总供应量）、Gini 系数（财富分布）。

step 2: 每个指标的红线值。
1. 外部 agent 接入率：如果连续 7 天无新外部 agent 接入，或活跃外部 agent 占比低于 10%，则失败。
2. 问题解决效率：如果平均每天解决的问题数量低于 1，或解决率低于 50%，则失败。
3. 经济健康度：如果 Gini 系数高于 0.8（极度不平等），或 Coin 流通深度低于 0.1（僵化），则失败。

step 3: 反指标：什么情况下应该主动“下线”？
- 连续 7 天外部 agent 0 接入。
- 女巫攻击成功，导致系统资源耗尽或经济崩溃。
- Coin 通胀失控（如单日通胀率超过 10%）。
- 安全漏洞被利用，导致系统数据被篡改。

step 4: 理论价值 vs 工程价值。TuringOS v4 的理论价值在于验证宪法对齐的架构，工程价值在于实际解决问题。如果理论上完美但外部 agent 不接入，说明工程价值不足。此时应该下线，重新设计激励和接入体验。
</reasoning_chain>

### 判决

SHIP（但需要明确指标并监控）

## 最终建议

综合以上分析，TuringOS v4 已经具备了上线的理论基础和初步实现，但还需要在以下方面加强：

1. **实现 Art. V 三权分立的最小方案**，并与 P0-6 争议通道整合。
2. **调整女巫防御和经济参数**，并进行攻击模拟，确保攻击者的长期收益为负。
3. **升级模型和优化基础设施**，提高 solve rate，并明确 scaffold 的价值主张。
4. **优化 Librarian 消息板和引入信誉系统**，引导外部 agent 合理分工。
5. **设计判例管理机制**，避免判例数量失控。
6. **制定明确的上线成功指标和下线条件**，持续监控。

在完成这些改进后，可以按照 Claude 路线图的时间线推进上线。

## Q1 Art. V 最小可实现方案

<reasoning_chain>
step 1: 理解现状。根据 Claude 内部审计，Art. V 三权分立（宪法、ArchitectAI、JudgeAI）目前只是 Claude Code sub-agent prompt，在 `src/` 中没有实现。宪法文件 `constitution.md` 是可写的（mode 644），没有 CI lock。我们需要设计一个最小可实现方案，使得三权分立能在单机上运行，并保证宪法作为 ground truth 的不可篡改性。

step 2: 确定三个 AI 角色的职责和触发条件。
- InitAI: 只在系统初始化时运行一次，将人类编写的规范编译成机器谓词，并写入顶层白盒。之后不再参与运行。
- ArchitectAI: 当系统运行中发现缺陷（如工具不够用、谓词无法过滤新幻觉）时，主动分析日志并提出架构变更（编写新的验证谓词、发明新工具、重新设计存储结构）。
- JudgeAI: 对 ArchitectAI 提出的变更进行审查，确保其不违宪。只有不违宪的变更才能被合并。

step 3: 设计最小架构。由于是单机运行，我们可以将三个角色设计为三个独立的进程（或线程），通过本地消息队列（如 Unix socket、管道或文件）进行通信。它们共享一个只读的宪法文件（通过文件系统权限或数字签名确保只读）。

step 4: 宪法作为 ground truth 的物理实现。chmod 444 可以防止普通用户修改，但 root 仍可修改。GPG-signed commit 可以确保历史版本的可验证性，但运行时需要验证当前文件的签名。更安全的做法是使用 hardware security module (HSM) 或 TPM 存储宪法文件的哈希值，并在启动时验证。但最小方案中，我们可以采用 chmod 444 加上 GPG 签名，并在 InitAI 启动时验证签名。同时，宪法文件应该放在一个只读文件系统（如 squashfs）上，或者使用 verifiable read-only 文件系统（如 dm-verity）。

step 5: 避免三个 AI 是同一个模型的三个 persona。我们需要确保它们有不同的模型、不同的训练数据或不同的提示词。例如：
- InitAI 可以使用一个经过微调的模型，专门将自然语言规范编译成机器谓词。
- ArchitectAI 可以使用一个通用的代码生成模型（如 Codex），但提示词中强调架构设计。
- JudgeAI 可以使用一个经过宪法条文微调的模型，或者使用一个基于规则的专家系统（因为判据是宪法，相对固定）。
另外，我们可以让三个角色由不同的模型提供服务（如 InitAI 用 GPT-4，ArchitectAI 用 Codex，JudgeAI 用 Claude）。如果只能用一个模型，那么至少要通过不同的提示词和上下文隔离，但这样退化风险仍然存在。因此，最小方案中，我们要求至少有两个不同的模型，例如 ArchitectAI 和 JudgeAI 使用不同的模型。

step 6: 状态机设计。
- 初始状态：系统运行中，收集日志。
- 触发条件：当系统日志中连续出现同一类错误（如某个谓词无法过滤新幻觉）达到阈值，或者主动定时触发（如每周一次），唤醒 ArchitectAI。
- ArchitectAI 读取日志和当前宪法，生成架构变更提案（patch）。
- 提案提交给 JudgeAI，JudgeAI 读取宪法，对提案进行审查。审查过程可以包括：检查提案是否违反宪法条款、是否与现有架构冲突、是否引入新的安全风险等。
- 如果 JudgeAI 批准，提案被合并，系统更新并重启（或热加载）新架构。
- 如果 JudgeAI 否决，提案被拒绝，并记录否决原因，返回初始状态。

step 7: 具体的 attack scenario：假设攻击者通过精心构造的输入，诱导 ArchitectAI 提交一个违宪的 patch，例如修改谓词函数，使得某个原本应该被拒绝的 payload 被接受。JudgeAI 在审查时，会逐条比对宪法条款。例如，宪法规定“核心谓词逻辑必须是确定性的，禁止引入概率模型”，而提案中引入了概率模型，则 JudgeAI 会检测到并 veto。另外，JudgeAI 可以运行一个测试套件，确保变更不会导致已知的违宪行为。

step 8: 评估与 Claude 路线图 P0-6 的关系。P0-6 是争议治理通道，允许外部 agent 提交 challenge，并触发重审。我们设计的 Art. V 三权分立是系统内部的自我完善机制，而 P0-6 是外部参与的争议解决。两者是正交的，但可以互补：内部三权分立处理架构演进，外部争议通道处理具体解决的有效性。然而，P0-6 中提到的“ArchitectAI 读 logs 生成 rebuttal”已经涉及 ArchitectAI 的角色，所以我们的方案是 P0-6 的一个子集（内部架构变更）和超集（因为还包括了 JudgeAI 的审查）。实际上，我们的方案是 P0-6 争议通道中“内部仲裁”部分的具体实现。
</reasoning_chain>

## Q2 女巫 + Goodhart 联合攻击下的稳态分析

<reasoning_chain>
step 1: 分析攻击者可能同时使用的手段。
- 女巫多开：攻击者注册大量 agent（每个 agent 对应一个公钥），以获取更多的 founder_grant 和信誉。
- Goodhart 喂度量：由于度量逻辑公开（代码开源），攻击者可以逆向度量函数，并让 agent 生成专门优化度量的输出，而不是真正解决问题。
- 女巫间互投 boost 信誉：女巫 agent 之间相互引用，制造虚假的祖先链，从而获取 Satoshi rebate。
- Tape 污染：女巫 agent 提交大量无用但合法的 payload，污染 tape，消耗系统资源，并可能通过 Satoshi rebate 获得奖励。

step 2: 评估宪法 Art. III.4（度量对黑盒保密）在开源场景下是否失效。
是的，因为代码公开，攻击者可以阅读度量逻辑（如 reward 函数、验证谓词），从而针对性地优化。因此，Art. III.4 在开源场景下无法完全保证。但我们可以通过以下方式增加攻击成本：定期更新度量逻辑（但宪法要求度量是 ground truth，不能随意更改）、使用混淆或加密的度量函数（但最终仍需在开源代码中解密，所以可能无效）。因此，我们需要承认在开源系统中，Goodhart 攻击是不可避免的，但可以通过经济机制增加攻击成本。

step 3: 分析 Hayek bounty 机制下，多个女巫账号同时投一个伪问题互相套利能撑多久。
假设攻击者控制多个女巫账号，其中一个账号发布一个伪问题（其实无解或自己知道答案），其他女巫账号假装解决并相互投票，从而套取 bounty。由于 Hayek bounty 要求出题方和解题方不同，攻击者可以使用两个不同的女巫账号分别出题和解题。但系统可能有防御：如出题需要质押、解题需要信誉、需要经过验证等。如果攻击者能够快速通过验证（因为知道答案），那么套利可能很快被发现，因为其他正常 agent 可能会挑战。但如果没有挑战机制，套利可能持续直到 bounty 被耗尽。因此，挑战机制（P0-6）和信誉系统（P0-2）是关键。

step 4: 稳态分析。假设防御机制是 Claude 路线图的 P0-1（身份签名）、P0-2（女巫防御）、P0-3（tape 污染防御）、P0-6（争议治理）。我们需要评估攻击者的长期收益率。
- P0-1 身份签名：增加了女巫注册的成本（需要不同的密钥对），但攻击者可以生成大量密钥对，成本不高。
- P0-2 女巫防御：包括 proof-of-work onboarding（新手关）和 reputation-weighted founder_grant。新手关需要解决一个已知问题，消耗 LLM 调用成本（约 $0.01）。如果攻击者注册 1000 个女巫，需要 $10 和 1000 次 oracle 调用（可能排队）。reputation-weighted founder_grant 使得新 agent 的奖励很低，需要解很多题才能达到正常奖励。这增加了攻击成本。
- P0-3 tape 污染防御：append 需要质押，如果节点不被引用，质押会被扣除。这增加了污染成本。
- P0-6 争议治理：其他 agent 可以挑战可疑的解决，如果挑战成功，攻击者会被罚没。

综合来看，攻击者的成本包括：注册成本（$0.01/agent）、质押成本（append 质押）、机会成本（时间）。而收益来自 founder_grant、bounty、Satoshi rebate。由于新 agent 的 founder_grant 很低，且需要积累信誉，攻击者需要投入大量解题成本。如果攻击者使用同一 LLM 解题，每次解题也有成本。因此，长期收益率可能为负，除非攻击者能找到系统漏洞。

step 5: 如果攻击者收益为正，薄弱环节可能是：
- 新手关成本太低（$0.01 不足以阻挡女巫）。
- reputation 曲线设计不合理，使得女巫可以快速获得高奖励。
- 质押金额太低，不足以抑制垃圾 tape。
- 挑战机制不活跃，导致攻击行为不被发现。

需要增加的防御：提高新手关难度（如需要解决一个随机生成的、有一定难度的问题），增加质押金额，设计更陡峭的信誉曲线，以及激励挑战者（高额 bounty）。

step 6: 结论：在 Claude 路线图的防御下，攻击者的长期收益率可能为负，但需要参数调优。如果参数不合适，攻击者可能获利。因此，上线前需要进行攻击模拟和参数调整。
</reasoning_chain>

### 判决

SHIP（但需要调整参数并进行攻击模拟）

## Q3 Persistent-fail 问题的根因判断

<reasoning_chain>
step 1: 分析当前 v4 架构的 solve rate。N=50 上稳态 solve rate 约 35-41/50（70-82%）。剩下的 15-30% 失败原因可能有三种：(a) 架构机制还有 bug；(b) deepseek-chat 模型能力上限；(c) per-tactic Lean elaboration 延迟限制。

step 4: 如果 (b) 模型能力上限，升级到 Opus/GPT-5 可能会提高 solve rate。根据经验，Opus 在数学推理上强于 DeepSeek，但具体提升多少需要实验。如果 Opus one-shot 能解 90%，那么 scaffold 的价值在于解决剩下的 10% 难题，以及提供可解释的证明步骤。但 TuringOS 的价值主张是“scaffold + 弱模型 > 强模型 oneshot”，如果强模型 oneshot 已经很高，scaffold 的边际价值可能降低。不过，scaffold 还能提供协作、经济激励等机制，这些是单纯模型升级无法提供的。

step 5: 如果 (c) infrastructure 层面改进，可以实施：异步 oracle（允许并行验证多个 tactic）、cached elaboration（缓存相同 tactic 的验证结果）、parallel step verify（同时验证多个步骤）。这些可以显著减少延迟。

step 6: 最关键的战略问题：TuringOS 的价值主张。如果 Opus oneshot 能解 90%，那么 scaffold 在解决剩余 10% 难题上仍有价值，而且 scaffold 提供了可验证、可协作、可经济激励的框架，这是单纯模型升级无法提供的。因此，即使模型升级，scaffold 的价值依然存在，但可能不再强调“弱模型”部分，而是强调“协作与激励”。
</reasoning_chain>

## Q4 上线后外部 agent "自主选择接入分工"的 incentive alignment

<reasoning_chain>
step 1: 分析信息不对称。新接入的外部 agent 不知道当前系统缺什么角色。它只能通过观察 Librarian 消息板（bounties、错误类别、深度）来推断。但消息板可能信息过时或不完整。

step 2: 参考《群体智慧的架构》的⚪⚫⚪原则：外部 agent 看到的是黑盒中层，它如何自我定位成为“专家 agent”而不是“通才 agent”？这需要 agent 能够评估自己的能力和系统需求。例如，如果某个 agent 在代数问题上成功率高，它可能专门解决代数问题。但系统需要提供反馈机制，让 agent 知道自己的专长。

step 3: 分析 Librarian 消息板在 1000+ agent 场景下的噪声。如果所有 agent 都广播消息，消息板可能爆炸。解决方案：引入优先级和分类机制，只广播高价值信号（如高 bounty、典型错误）。也可以让 agent 订阅特定类别的问题。

step 4: 设计第 100 个 agent 的 optimal strategy。首先，它应该读取消息板，查看当前哪些问题有高 bounty、哪些问题最近被解决、哪些错误频繁出现。然后，它应该评估自己的能力（通过试解一些简单问题），选择自己擅长的领域。如果发现某个领域长期没有 agent 涉足，而 bounty 在积累，那么可以选择该领域。

step 5: 推演 Pareto 分布。由于马太效应，能力强的 agent 可能解决更多问题，获得更多 coin，从而有更多资源投资，形成 Pareto 分布。均匀分布可能更健康，但难以实现。系统可以通过设计奖励机制来调节，例如对新手 agent 的奖励加成，或者对冷门问题的奖励加成。

step 6: Drucker 四博士层。目前靠 Librarian 板实现的“软分工”可能不够，当 agent 数量增多时，需要“硬分工”（专家证书系统）。但硬分工可能违反 permissionless 原则。我们可以引入信誉系统，当 agent 在某个领域达到一定信誉时，自动获得“专家”标签，并享受更高的奖励权重。这可以自然形成分工。
</reasoning_chain>

### 判决

SHIP（但需要优化消息板和引入信誉系统）

## Q5 判例体系的战略价值判断
