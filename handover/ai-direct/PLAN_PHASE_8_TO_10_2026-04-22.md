## § 2. Phase 8: BLOCKER 修复

### 8.A — `bus.snapshot()` 空 balances 修复（**最紧急**）

- **依据**：Codex N-2（`src/bus.rs:567-570`）
- **现状**：`snapshot()` 硬编码 `balances: HashMap::new(), portfolios: HashMap::new()` → agent prompt 永远看到 Balance=0
- **假说**：这是历次 TAPE_ECONOMY / Hayek bounty 实验失败的主因（不是激励曲线问题）
- **Spec**：
  - `bus.snapshot()` 必须从 `self.tools` 里找到 `WalletTool` 并真实 enumerate balances
  - `portfolios` 同理，从 `self.kernel.markets` + 每 agent 的 `Position` 构建
  - 输出结构：`balances: HashMap<agent_id, f64>`, `portfolios: HashMap<agent_id, Vec<(market_id, shares)>>`
  - 性能考虑：snapshot 被每 tick 调用，若 agent×market 很多需 cache
- **Test 覆盖**：新增 `tests/snapshot_nonempty.rs`：
  1. init → genesis → snapshot balances 非空且等于 founder grant
  2. invest → snapshot portfolios 包含 position
  3. halt_and_settle → snapshot balances 含 payout
- **Acceptance**：
  - 所有现有 test 通过
  - 新测试 3 条全绿
  - 跑 N=5 smoke：prompt 里 `Balance: X Coins` 必须 X > 0（过去永远 0）
- **红线检查**：不触发 7 条宪法红线中任一条
- **分支**：`feat/phase-8-snapshot-fix`

### 8.B — `oneshot` 路径强制走 C-043 mandatory wtool

- **依据**：Codex N-1（`evaluator.rs:108-143,196-215`）
- **现状**：oneshot 直接 `oracle.verify_omega` 返回 PputResult，不经 bus write
- **违反**：宪法 `constitution.md:521-524` `∏p=1 → wtool → Q_{t+1}` + `cases/C-043`
- **Spec**：
  - oneshot 通过后必须调 `bus.append_oracle_accepted(agent_id, payload, parent=None)` 把 proof 写入 tape
  - WAL 持久化（C-037 要求）
  - 保持 oneshot 的单次延迟预期（不要因为 wtool 加 50% latency）
- **Test**：新增 `tests/oneshot_writes_tape.rs`：oneshot 成功后 `kernel.tape.time_arrow()` 长度 +1
- **Acceptance**：
  - N=20 oneshot A/B: solve rate Δ ≤ 1（新路径不应降解题）
  - tape 持久化验证：WAL 文件有 oneshot proof node
- **红线检查**：红线 #3（raw CoT to tape）必须持续绿
- **分支**：`feat/phase-8-oneshot-wtool`（与 8.A 可并行）

### 8.C — `append_oracle_accepted` 加 capability token

- **依据**：Codex V-1（`src/bus.rs:174-176`）
- **现状**：public 函数 + oracle_blessed 调用者自主声明
- **Spec**：
  - 定义 `pub struct OracleReceipt { payload_hash: [u8;32], verdict: Verdict, oracle_id: Uuid, signature: Option<[u8;64]> }`
  - `verify_omega_detailed` / `verify_partial` 返回 `(Verdict, OracleReceipt)`
  - `append_oracle_accepted(agent_id, payload, parent, receipt)` 要求传 receipt
  - bus 内部校验 `receipt.payload_hash == hash(payload)` + `receipt.verdict == Complete/PartialOk`
  - MVP 阶段 signature 可为 None（单进程），Phase 10 外部接入时必须填 Ed25519
- **Test**：`tests/oracle_receipt_required.rs`：
  1. 伪造 receipt（hash 不匹配）→ `append_oracle_accepted` 返回 Err
  2. 合法 receipt → Ok
  3. 无 receipt → 编译错误（API 签名强制）
- **Acceptance**：
  - 现有全部 test 编译通过（需要 migrate 调用点）
  - 新 test 3 条全绿
  - `append_oracle_accepted` 在 grep 全部调用点都传 receipt
- **红线**：#6 ∏p 可重跑验证 — receipt 里的 payload_hash 和 verdict 就是"re-verify 凭证"
- **分支**：`feat/phase-8-oracle-receipt`（依赖 8.B，因为 oneshot 也要传 receipt）

### 8.D — `decide`/`omega` 进 forbidden_patterns 完整执行 C-011

- **依据**：Codex N-3（`lean4_oracle.rs:13-20,57-88`）+ `cases/C-011`
- **现状**：只 ban `native_decide`，`decide` 和 `omega` 未禁，测试 `lean4_oracle.rs:395-399` 还显式 assert `decide` 允许
- **Spec**：
  - `FORBIDDEN_PATTERNS` 加入 `\bdecide\b` 和 `\bomega\b`（用 regex 边界，避免误伤 `native_decide` 已在列表、`omega_wtool` 等 identifier）
  - 更新测试：`tests/oracle_c011.rs` assert 三者都被拒
  - **潜在冲突**：Mathlib 里合法使用 `decide` 的 theorem 名称（如 `Decidable.decide`），需 whitelist
- **Risk 评估**：可能拦截一些历史上合法的 solve。**先跑影响分析**：grep N=50 past logs 里 `decide`/`omega` 出现频率
- **Test**：`tests/oracle_c011_complete.rs`
- **Acceptance**：
  - 影响分析：past logs 里如果 >5 solves 用了 bare `decide` 则需 review 每条是否真的是 brute force
  - N=20 smoke：新 FORBIDDEN 未把正常 solve rate 拉低 >10%（若 >10% 则需 refine regex 边界）
- **红线**：#6 ∏p 一致性 — 三家外部模型独立跑 forbidden check 结果相同
- **分支**：`feat/phase-8-c011-complete`

## § 3. Phase 9: 统计可信基线

### 9.A — 4 seeds × N=50 dual-mode 实验

- **依据**：Gemini Q6 power analysis
- **动机**：LATEST.md Next Step #1 的"一次 N=50 dual-mode"**统计无效**（power 46%）
- **Spec**：
  - seed 池：`{74677, 31415, 2718, 141421}`（pre-registered，不允许中途换）
  - 每 seed 跑一次 N=50 dual-mode（`TURING_STEP_ONLY=0`，step + complete 都可选）
  - 每次跑前 smoke probe（`feedback_smoke_before_batch`：mathd_algebra_148 oneshot 30-60s）
  - **不 paired**（配对需要同问题两种 treatment，这里是同 treatment 不同 seed）
  - 产出：4 × jsonl 日志 + 每次的 tool_dist / gp_node_histogram / timeout count
- **分析**：
  - 聚合 solve rate = (x1+x2+x3+x4) / 200
  - 95% CI via Wilson score interval
  - **判据**：CI 下界 ≥ 0.70 → 宣称 "dual-mode solve rate ≥ 70%"；若 CI 下界 < 0.70 → 只能说 point estimate
  - **vs. monolithic**：若 CI 上界 < 0.85 → 显著低于 monolithic；若 CI 含 0.85 → equivalent
- **Acceptance**：报告 = CI 数值 + 是否能宣称"dual 恢复到 monolithic 水平"
- **成本**：4 × ~6h wall × ~$30 ≈ 24h / $120

### 9.B — DAG 涌现 CI-verification

- **依据**：Gemini Q2 CHALLENGE
- **Spec**：
  - 同样 4 seeds 的 step-only 条件（`TURING_STEP_ONLY=1`）N=50
  - 记录每 seed 的 gp_node_histogram
  - **判据**：4 次独立 run 中至少 3 次出现 depth ≥ 10 的 DAG 节点 → "emergent depth 稳定复现"
  - 若 3/4 出现 → PASS；2/4 → 需要加跑 seed 到 6 次；≤ 1/4 → FAIL（Phase 7 depth 23 是 lucky seed）
- **可复用 9.A 的 infra**（只是 env var 不同）
- **成本**：再 4 × ~6h ≈ 24h / $120

### 9.C — Law 2 守恒测试补齐（invest refund + Hayek bounty）

- **依据**：Gemini Q1
- **Spec**：
  - `tests/reward_pull_conservation.rs` 加两个测试：
    1. `test_invest_refund_conservation`: invest → kernel 失败 → credit refund → Σ debit == Σ credit
    2. `test_hayek_bounty_payout_conservation`: init bounty → halt_and_settle → Σ bounty_payouts == bounty_lp_seed
  - 使用 property-based test（proptest）随机生成 tx 序列验证 invariant
- **Acceptance**：新增 2+ 测试全绿，total_supply invariant 在 1000 random tx 下保持

### 9.D — Karpathy 优化 micro-benchmark

- **依据**：Gemini Q3 + Codex Q4 CHALLENGE
- **Spec**：只对 3 条候选做 micro-benchmark：
  1. `trace_ancestors` HashSet → Vec<&str>（DAG 深度 ≤ 100 时）
  2. `author.to_string() + payload.to_string()` → `Arc<str>`
  3. `graveyard + TopKClasses` 双写 → 单 counter
- **方法**：criterion.rs micro-benchmark，对比 before/after 的 wall time + heap allocs
- **判据**：每条 > 5% 改进 → 实施；< 5% → 记入 "不优化决策档案"
- **不做**：其他 7 条（InvestOnly clone / Box<dyn> downcast / WAL clone 等）先搁置

### Gate 9 → 10

**必过项**：
1. 9.A dual-mode CI 报告成文（规范陈述 CI 范围，不 over-claim）
2. 9.B DAG 涌现：3/4 seeds 有 depth ≥ 10 DAG → PASS；否则加跑或 retract

**W-A.2 Sigmoid γ 曲线女巫防御（P0-2）**
- Gemini spec: `γ(n) = 0.05 / (1 + exp(-0.146 * (n - 25)))`
- `WalletTool::on_agent_register` 时用 γ(n_solved) × FOUNDER_GRANT_BASE 发放
- N=20 A/B vs 当前 flat γ：solve rate 不降；女巫收益模拟（单机跑 1000 账号 simulator）预估 O(1)
- PoW 新手关：新 agent 注册时需先解一个 easy benchmark（`mathd_algebra_44`）才分配 wallet

**W-A.3 Tape 污染 + 园丁 GC（P0-3 + C-045）**
- tombstone 机制：`Node { state: Active | Tombstone }` enum；GC agent 每 N 秒扫老 node（age > 24h + citations == 0）打 tombstone
- tombstone 节点不进 snapshot.tape，但 WAL 保留
- append 需 stake ε Coin，节点被 tombstone 时 stake 没收
- Test：1000 append + 24h 推进 + 无引用 → 全部 Tombstone；WAL 保留

### Wave B: 经济 + Oracle 基础设施（P0-4/P0-5/P0-5b）

**W-B.2 Fingerprint cache（P0-5）**

- 实际测量：跑一次 N=50 记录 hit rate，若 < 15% 则重评成本

## § 5. 自主执行协议

### 工作节拍
- **每工作日 1 个 sub-task**（不赶进度，按 feedback_phased_checkpoint "逐项改进逐项测试"）
- **每 sub-task 完成后 commit + 更新 AUTO_RESEARCH_NOTEPAD** F-2026-04-xx-yy 条目
- **每 Wave 完成后强制 CHECKPOINT_WAVE_X doc + 7 红线自查 + auto-pause 等人审**

### 失败/回退
- sub-task N=20 A/B solve rate 降 >10% → 自动回退到 parent branch
- dual-audit 任一 VETO → 停，改 spec，再试；不可为了"过关"妥协
- 连续 3 个 sub-task 失败 → 整个 Wave 暂停，召集人审

### 资源 budget
- Phase 8: ~3d × $40/d = $120
- Phase 9: ~$240（4×dual-mode + 4×step-only）+ ~1d dev
- Phase 10: ~7d × $60 = $420 + 长尾外部测试
- **累计硬顶 $1000**；超 $700 必须人审

### 暗礁清单（遇到主动停）
- 任何修改 `constitution.md` 的 PR（只能人手 + GPG）
- 任何修改 `bus.rs` / `kernel.rs` / `wallet.rs` 的 PR（`STEP_B_PROTOCOL` 强制 parallel branch）
- 发现新的宪法违规（N-1~N-3 级别）→ 立 judicial case 再前进
- 外部 agent 首次接入出现 anomalous behavior（非预期 tool use / 超预算 tx）
