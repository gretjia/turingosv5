## § 1. Objective

Current state (Phase 8 + Phase Z): ~85% of flowchart elements have code analogs but only partial evidence of 1-to-1 mapping. User's concern: "每一行代码都有效，都对齐了宪法".

Goal: **verifiable** constitutional alignment where
1. Every flowchart element has a unique code symbol (file:line).
2. Every non-trivial code block has a doc comment citing its constitutional origin.
3. A traceability matrix lets auditors go **both** directions (spec → code, code → spec).
4. Conformance tests exercise every element (not just happy-path).
5. One medium-difficulty real MiniF2F problem validates end-to-end behavior.

## § 2. Industry best-practices reference

| Practice | Used where | Application here |
|---|---|---|
| **Literate programming** (Knuth) | TeX, Axiom | doc comments cite flowchart node; single source of truth |
| **Traceability matrix** (DO-178C / ISO 26262) | avionics, auto | spec element ↔ code ID ↔ test ID, audited |
| **Architecture Decision Records (ADR)** (Nygard) | many Rust projects | per-decision markdown referencing constitution |
| **Model-Driven Engineering (MDE)** | UML → code | mermaid = model; Rust = generated/hand impl; conformance check |
| **Specification-by-example** (Gojko Adzic) | BDD tests | each flowchart edge has an executable scenario |
| **Bidirectional mapping** (Horrocks, MBSE) | sysml | orphan-code + orphan-spec detection |
| **Type-driven design** (Brady) | Idris, Rust | encode constraint in type system (can't forget at API level) |
| **Property-based testing** (QuickCheck) | proptest | laws (Law 1/2) as properties, not examples |

Adopted combination: **traceability matrix (DO-178C-style) + doc-comment backlinks (literate) + conformance tests (BDD-ish) + property tests for laws (QuickCheck)**.

## § 5. Work plan

### Stage 1 — Audit + Matrix Build (no code changes)
- **Agent A (Claude Code)**: scan constitution.md for all 3 flowcharts, build raw element list (markdown table, 45+ rows)
- **Agent B (Codex)**: code-scan pass — for each element, grep candidate Rust symbols, propose file:line mappings
- **Agent C (Claude Code)**: cross-check Codex's proposals; flag conflicts + orphans (code without spec, spec without code)
- Output: `TRACE_MATRIX_v0_2026-04-22.md` with status column per row: ✅ exact / ⚠️ partial / ❌ missing / 🔍 orphan

**Budget**: ~2h, ~$5 LLM.
**Gate**: I present matrix to user. User reviews ⚠️/❌ rows, marks which to fix vs accept-with-caveat.

### Stage 2 — Doc-Comment Backlinks (mechanical)
- **Agent D (Claude Code)** in isolated worktree: for every ✅/⚠️ row, add or update doc comment citing the flowchart element, e.g.:
  ```rust
  /// FC-1 `rtool`: Art. IV mermaid projection of `Q_t` → `⟨q_i, s_i⟩`.
  /// Corresponds to: TRACE_MATRIX row R-12.
  pub trait ReadTool { ... }
  ```
- No behavior change; doc-only.
- Include a `#[doc = "..."]` CI lint (procedural macro OR manual grep) that fails if any trace-matrix-listed symbol lacks its backlink.

**Budget**: ~3h, ~$3 LLM.
**Gate**: `cargo check` + `grep -c "TRACE_MATRIX" src/` matches expected count.

### Stage 3 — Fill ❌ Missing Elements (code changes)
- For each ❌ row in the matrix (expected < 10):
  - **Agent E (Claude Code)** if element is local (SDK/evaluator)
  - **Agent F (Codex)** if element touches bus/kernel (restricted)
  - Each fix: 1 branch in worktree → doc-backlink + implementation + conformance test
- Sequential merging to avoid conflict (lessons learned from today's 3-agent merge).

**Budget**: ~4h, ~$10 LLM.
**Gate**: all tests pass + trace matrix 100% ✅ or ⚠️-with-accepted-caveat.

### Stage 4 — Conformance Test Battery
- **Agent G (Claude Code)** writes one test per matrix row (if not already covered). Target files:
  - `tests/fc1_cycle_conformance.rs` (~20 tests)
  - `tests/fc2_init_halt_tick_conformance.rs` (~15 tests)

- **Property tests** via `proptest` for Law 1 (append-free) and Law 2 (conservation).

**Budget**: ~3h, ~$3 LLM.
**Gate**: all conformance tests green; property tests 10K runs green.

### Stage 5 — Medium-Difficulty Real-Problem Validation

## § 7. Orphan-code + orphan-spec detection

Bidirectional sweep (Stage 1 parallel step):
- **Orphan code**: Rust files grep for `pub fn` / `pub struct` / `pub trait`. Any symbol not in TRACE_MATRIX is an orphan. Options: (a) add row, (b) justify as internal plumbing, (c) delete if truly unused.
- **Orphan spec**: flowchart elements without code candidates. Options: (a) implement in Stage 3, (b) explicitly mark as **Phase 11+ out-of-scope** in matrix with justification.
