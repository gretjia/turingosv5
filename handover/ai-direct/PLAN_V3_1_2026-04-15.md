### Change 7 [C] — Interleaved execution
**Protocol**: For each problem P in the 50-tuple, run condition order rotated:
- Problem 1: oneshot → n1 → n3
- Problem 2: n1 → n3 → oneshot
- Problem 3: n3 → oneshot → n1
- Problem 4: oneshot → n1 → n3 (repeat)

Rotation cycle of 3 ensures each condition is equally distributed across early/middle/late API-call positions. API drift (rate-limit variance, etc.) averages out.

### Change 8 [G+C] — Precise decision rules
**For pairwise SolveRate comparison** (A vs B on N=50):
- **Strict win**: `|solves(A) - solves(B)| ≥ 3` (≥6 pp, ~2σ under binomial noise)
- **Practical equivalence**: `|solves(A) - solves(B)| ≤ 1` (≤2 pp)
- **Gray zone**: gap of 2 → reported as "marginal, not conclusive"

**Tie-break** (if strict win): lower Aggregate_PPUT loses only if primary tied; otherwise primary governs.

**Conflict rule**: primary (SolveRate) always governs. Secondary (Aggregate_PPUT) reported, never overrides.

### Change 9 [C] — Paired-subset analysis
**Definition**: Problems where all 3 conditions emitted a PPUT_RESULT (not abort-truncated, not MEASUREMENT_ERROR).
**Primary analysis**: On all 50 (aborts = failures).
**Secondary analysis**: On paired subset (fair apples-to-apples).
**Both reported** in `frozen_analysis.py` output.

