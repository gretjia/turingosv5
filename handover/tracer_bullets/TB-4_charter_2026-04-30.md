## 0. One-line goal

Activate the **second** half of the RSP "submitted → verified → challenge-window-open" admission spine: fill the existing `Verify` and `Challenge` arms of `dispatch_transition` (currently `NotYetImplemented` stubs at `src/state/sequencer.rs:324-325`) so that

- accepted `VerifyTx` debits `balances_t[verifier_agent]` by `verify.bond` and credits `stakes_t[verify.tx_id]` (verifier puts skin in the game),
- accepted `ChallengeTx` debits `balances_t[challenger_agent]` by `challenge.stake` (the **NO** position) and credits `challenge_cases_t[challenge.tx_id]` (challenger publishes counter-evidence with bonded skin),
- both arms route bad admissions to L4.E without mutating economic_state,
- the **challenge-window structure** is emitted via `ChallengeCase.opened_at_round = q.q_t.current_round` — the structural anchor RSP-3 needs to compute "window closed". **TB-4 ships the anchor; RSP-3 ships the closure + slash. These three layers (RSP-2 admission / RSP-3 closure+slash / RSP-4 settlement+attribution) are not collapsible.**

`VerifyTx` and `ChallengeTx` are **not** new variants; they have been first-class typed schemas at `src/state/typed_tx.rs:240-267` since CO1.1.4-pre1 and were excluded from TB-2 / TB-3 production scope only because their dispatch arms were stubbed. TB-4 lights up those arms.

## 1. roadmap_exit_criteria_addressed

- **P3:4** — `challenge_tx must lock NO stake` — **fully discharged** through the formal admission gate: `ChallengeTx` is accepted only when `challenge.stake.micro_units() > 0` AND `balances_t[challenger_agent] >= challenge.stake`; on accept, balance is debited by `challenge.stake` and an entry inserted into `challenge_cases_t` with `bond = challenge.stake`. Rejected ChallengeTx routes to L4.E with no economic-state mutation. (P3:4 was RED through TB-3.)
- **§ 3 P3 Forbidden — "verifier 无责任盖章"** — **structurally discharged** via `VerifyTx` accept = balance debit + bond locked into `stakes_t[verify.tx_id]`. After TB-4 a verifier cannot stamp `Confirm` without bonded exposure; the bond stays locked through the implicit challenge window (`opened_at_round` anchored on the corresponding ChallengeCase or work-tx liveness). Slashing the bond on a wrong verdict is RSP-3.
- **§ 3 P3 Forbidden — "ghost liquidity / per-node automatic injection"** — re-affirmed: every `stakes_t` insert from VerifyTx and every `challenge_cases_t.bond` insert from ChallengeTx is paired with an explicit `balances_t` debit in a CTF-conserved transition. No new liquidity source is introduced.
- **Partial structural progress on P3:6** ("predicate-pass yields only provisional payment") and **P3:7** ("settlement only after challenge window closed and no valid challenge"): RSP-2 emits the `opened_at_round` anchor that RSP-3 needs to compute window closure. RSP-2 itself does **not** flip P3:6 or P3:7 GREEN — full discharge requires `provisional_accept_tx` + `challenge_resolve_tx` + `settlement_tx` (RSP-3 / RSP-4).

Per directive § 3 + ROADMAP § 11 priority rule ("lowest-numbered phase with a RED kill criterion wins"), **P3:4** is the lowest-numbered P3 RED criterion through TB-3; TB-4 closes it. P3:9 (失败 Solver slash) requires actual slash execution and stays RED through RSP-3.

### 3.2 9-sub-field EconomicState invariant — preserved

TB-3 charter § 5 #9 forbids new EconomicState sub-fields. TB-4 honors this **strictly**:

- Verifier bonds are tracked **inside the existing `stakes_t: BTreeMap<TxId, StakeEntry>`** index. The natural key is `verify.tx_id` (the VerifyTx's own id), with `staker = verify.verifier_agent` and `task_id` looked up via the target work_tx's task_id (which is replay-deterministic from Q_t: `stakes_t[work.tx_id].task_id` already carries this binding per TB-3 § 3.4).
- Challenger NO stakes are tracked **inside the existing `challenge_cases_t: BTreeMap<TxId, ChallengeCase>`** index — this index already exists at `src/state/q_state.rs:333` exactly for this purpose. The natural key is `challenge.tx_id` (the ChallengeTx's own id), with `bond = challenge.stake.coin` (NO position amount), `challenger = challenge.challenger_agent`, `opened_at_round = q.q_t.current_round`.
- The 9 sub-fields of `EconomicState` are unchanged: balances + escrows + stakes + claims + reputations + task_markets + royalty_graph + challenge_cases + price_index.
- Conservation count remains **5 holdings**: balances + escrows + stakes + claims + challenge_cases.bond.
- Verifier verdict (`Confirm | Doubt`) is **NOT** stored in Q_t; it is recoverable from canonical L4 (the `VerifyTx.verdict` field of the L4 row). RSP-3's settlement engine will index it from L4 replay; no Q_t mutation needed in TB-4.

### 3.4 VerifyTx admission — what "verifiable" means in RSP-2 minimum scope

A `VerifyTx` is admissible iff:

1. `verify.parent_state_root == q.state_root_t` (StaleParent gate; per § 3.8 + Q2 schema bump).
2. `verify.bond.micro_units() > 0` (verifier puts skin in the game) — else `BondInsufficient`.
3. `verify.target_work_tx` exists in canonical L4 — else `TargetNotFound` (per § 3.8).
4. `verify.target_work_tx` is **present in `stakes_t`** (target's YES stake still locked = target still live and verifiable) — else `TargetWorkInactive` (per § 3.8).
5. `balances_t[verify.verifier_agent] >= verify.bond.coin` (verifier solvent) — else `InsufficientBalance`.
6. **No idempotency dedup** (per directive § 4 Q1 DEFER): TB-4 deliberately permits a verifier to submit multiple VerifyTx against the same target. A verifier may submit `Doubt` then later `Confirm` as evidence evolves; multiple distinct VerifyTx tx_ids each carry their own bond debit. Anti-spam / dedup mechanisms are RSP-3+ reputation-driven; embedding hard dedup in RSP-2 would foreclose legitimate evidence-evolution flows.

On accept:

```text
balances_t[verify.verifier_agent] -= verify.bond.coin
stakes_t.insert(verify.tx_id, StakeEntry {
    amount: verify.bond.coin,
    staker: verify.verifier_agent,
    task_id: stakes_t[verify.target_work_tx].task_id.clone()
})
state_root_t = sha256(VERIFY_ACCEPT_DOMAIN_V1 ‖ prev ‖ canonical_hash(verify))
```

Per § 3.10: **the `verdict` field rides canonical L4 only**. Q_t is never mutated based on Confirm/Doubt; the verdict is a signed economic signal, not subjective truth.

### 3.5 ChallengeTx admission — what "challengeable" means in RSP-2 minimum scope

A `ChallengeTx` is admissible iff:

1. `challenge.parent_state_root == q.state_root_t` (StaleParent gate; per Q2 schema bump).
2. `challenge.stake.micro_units() > 0` — else `StakeInsufficient` (existing variant).
3. `challenge.target_work_tx` exists in canonical L4 — else `TargetNotFound`.
4. `challenge.target_work_tx` is present in `stakes_t` (still live) — else `TargetWorkInactive`.
5. `balances_t[challenge.challenger_agent] >= challenge.stake.coin` — else `InsufficientBalance`.
6. `challenge.counterexample_cid != Cid::ZERO` — else `EmptyCounterexample` (per directive Q7: distinct from MalformedPayload / PolicyViolation; needed by P4 cluster signal).
7. **No idempotency dedup**, AND multi-challenger explicitly legal (per directive Q4): distinct challengers MAY submit distinct ChallengeTx against the same `target_work_tx`; each opens a distinct `challenge_cases_t` row. Same challenger MAY submit distinct ChallengeTx against the same target with different `counterexample_cid` (evidence evolution; same Q1 reasoning as VerifyTx).

On accept:

```text
balances_t[challenge.challenger_agent] -= challenge.stake.coin
challenge_cases_t.insert(challenge.tx_id, ChallengeCase {
    challenger: challenge.challenger_agent,
    bond: challenge.stake.coin,
    opened_at_round: q.q_t.current_round,         // ← challenge-window structural anchor
    target_work_tx: challenge.target_work_tx,
})
state_root_t = sha256(CHALLENGE_ACCEPT_DOMAIN_V1 ‖ prev ‖ canonical_hash(challenge))
```

Per § 3.9: `opened_at_round` is a structural **anchor**, not a deadline check. RSP-2 never reads it back to gate behavior.

### 3.6 No state-machine status field on WorkTx

Per directive § 6: TB-4 does **NOT** add a `status: WorkStatus` field anywhere in Q_t.

ROADMAP § 3 P3 sketches a 7-state machine `OPEN → SUBMITTED → VERIFIED → PROVISIONAL_ACCEPTED → CHALLENGE_WINDOW → FINALIZED → PAID`. Liveness is read **implicitly**:

- `stakes_t.contains_key(&work.tx_id)` ⟹ work is in `SUBMITTED` (still live; YES stake still locked).
- `challenge_cases_t` rows whose `target_work_tx == work.tx_id` ⟹ work has open challenges (`CHALLENGE_WINDOW` open; multiplicity allowed).
- Absence from `stakes_t` ⟹ work has been resolved (RSP-3 territory; either finalized → stakes released, or slashed → stakes burned). RSP-2 never produces this state.

Pre-existing `TxStatus` enum (typed_tx.rs:200-206) is **runtime book-keeping only** (D-1 elision; never serialized into a TypedTx variant's wire bytes). Extending it into a Q_t field would conflate runtime book-keeping with state shape. Forbidden.

### 3.7 Slashing remains 100% out of scope

TB-4 does **not** execute any slash. Specifically:

- A wrong verifier verdict is not slashed in TB-4. The bond stays locked in `stakes_t[verify.tx_id]`. RSP-3's `ChallengeResolveTx` will read `verify.verdict` from L4 + `challenge_cases_t.target_work_tx == verify.target_work_tx` to decide the slash.
- A failed challenger (challenge fails post-resolution) is not slashed in TB-4. Bond stays in `challenge_cases_t[challenge.tx_id]`.
- A challenge-resolution win does NOT release the verifier bond or challenger bond in TB-4. RSP-3 introduces `ChallengeResolveTx` (system-emitted) which is the only legal mutator of those entries.

This is the **WP § 18 Invariant 6 + 7 boundary**: predicate pass produces (in the eventual P3-final state) only provisional reward; settlement requires challenge-window-closed-and-no-valid-challenge. TB-4 ships the "open" half; TB-5 (RSP-3) ships the "close" half.

### 3.8 Three-class error taxonomy (NEW per directive § 4 Q3)

Failure modes for VerifyTx / ChallengeTx admission are split into **three distinct TransitionError variants**, NOT folded into a generic `PolicyViolation`. This satisfies the constitutional principle: "错误类型应成为明确、低维、可统计的布尔/分类信号" (信号量化 / Art. I.2 + WP § 7-9).

| Variant | Semantics |
|---|---|
| `TargetNotFound` | `target_work_tx` does not exist in canonical L4 (no row with this TxId was ever accepted as a WorkTx). Distinguishes "you addressed nothing" from "you addressed something inactive". |
| `TargetWorkInactive` | The target WorkTx exists in L4 but is not in a verifiable / challengeable state — i.e., not present in `stakes_t` (either was never accepted, or has been resolved/finalized in a future RSP-3 path; in TB-4 minimum scope this collapses to "never accepted as a live WorkTx"). |
| `TargetNotVerifiable` | The target tx exists but is not structurally a WorkTx (e.g., target_work_tx points at a TaskOpenTx or EscrowLockTx tx_id). **Reserved**; in TB-4 minimum scope this is unreachable because L4 lookup by TxId would already produce a typed mismatch — but the variant is added for forward-compat completeness and named as a documented carve-out for RSP-3 (when finalized/expired-WorkTx liveness becomes a richer signal). |

These three are P4 Information Loom signals and MUST NOT be folded into one another. `MalformedPayload` (deserialization failure) and `PolicyViolation` (catch-all) remain distinct from these three (per directive Q7 reasoning extended to admission errors).

### 3.9 Challenge window — open-only-anchor rule (NEW per directive § 5.4)

TB-4 emits the challenge-window structural anchor:

```text
opened_at_round = q.q_t.current_round
target_work_tx = ...
```

TB-4 does **NOT**:

- compute `q.q_t.current_round - opened_at_round >= window_length` (no closure check),
- introduce a `challenge_window_length` parameter or constant,
- emit a `challenge_window_closed_tx` or any deadline-triggered tx,
- reject late-window VerifyTx / ChallengeTx (no temporal admission predicate),
- auto-finalize on window close,
- iterate `challenge_cases_t` to find expired windows.

All closure / resolution / finalization is RSP-3 territory. The anchor is **structurally sufficient** for RSP-3 to compute deadline math; nothing else is built in TB-4.

### 3.10 VerifyTx is signal+stake, not subjective judge (NEW per directive § 5.2)

The `VerifyVerdict` field (`Confirm | Doubt`) is preserved in the wire schema and rides canonical L4. Three explicit non-actions:

1. **Q_t is never mutated based on `verdict`.** A verifier's `Doubt` does NOT flip a status field, does NOT decrement reputation, does NOT change `stakes_t[target]`. It is a signed economic signal embedded in the L4 row, period.
2. **The verdict is not the final truth.** Final truth is determined by the predicate bundle (already evaluated at WorkTx admission), the challenge window outcome (RSP-3), and the settlement rule (RSP-3/4). A `Confirm` verdict from a verifier holding a bonded position is a market signal, not a constitutional ruling.
3. **顶层白盒不做主观审阅**. Per WP § 3.1, the top white-box runs deterministic statistics + predicates; it does not read narrative verdicts. The verifier's verdict is an opaque string-with-bond from the white-box's perspective; consumers (P4 Information Loom, RSP-3 settlement) read it from L4 by their own deterministic rules.

ChallengeTx by symmetry: submission ≠ slash; submission ≠ challenge upheld; submission ≠ verifier penalized. RSP-3 alone resolves these.

### 4.3 dispatch_transition arms — design

All arms remain pure (no I/O); apply_one writes the canonical L4 row. Three-class error taxonomy (§ 3.8) embedded:

```text
Verify arm (NEW; replaces NotYetImplemented at sequencer.rs:324):
  1. parent_state_root match else StaleParent.
  2. verify.bond.micro_units() > 0 else BondInsufficient.
  3. canonical-L4 lookup: target_work_tx ∈ accepted L4 rows? else TargetNotFound.
  4. liveness lookup: target_work_tx ∈ stakes_t? else TargetWorkInactive.
  5. balances_t[verify.verifier_agent] >= verify.bond.coin else InsufficientBalance.
  (NO idempotency dedup — directive Q1 DEFER.)
  6. q_next:
       balances_t[verify.verifier_agent] -= verify.bond.coin
       stakes_t.insert(verify.tx_id, StakeEntry {
           amount: verify.bond.coin,
           staker: verify.verifier_agent,
           task_id: stakes_t[verify.target_work_tx].task_id.clone()
       })
  7. assert_no_post_init_mint + assert_total_ctf_conserved(q, q_next, &[]) (debit = credit).
  8. q_next.state_root_t = sha256(b"turingosv4.verify.accept.v1" ‖ prev_root ‖ canonical_hash(tx)).

Challenge arm (NEW; replaces NotYetImplemented at sequencer.rs:325):
  1. parent_state_root match else StaleParent.
  2. challenge.stake.micro_units() > 0 else StakeInsufficient (existing variant).
  3. canonical-L4 lookup: target_work_tx ∈ accepted L4 rows? else TargetNotFound.
  4. liveness lookup: target_work_tx ∈ stakes_t? else TargetWorkInactive.
  5. balances_t[challenge.challenger_agent] >= challenge.stake.coin else InsufficientBalance.
  6. challenge.counterexample_cid != Cid::ZERO else EmptyCounterexample.
  (NO idempotency dedup; multi-challenger explicitly allowed — directive Q1 + Q4.)
  7. q_next:
       balances_t[challenge.challenger_agent] -= challenge.stake.coin
       challenge_cases_t.insert(challenge.tx_id, ChallengeCase {
           challenger: challenge.challenger_agent,
           bond: challenge.stake.coin,
           opened_at_round: q.q_t.current_round,         // ← § 3.9 anchor
           target_work_tx: challenge.target_work_tx,
       })
  8. assert_no_post_init_mint + assert_total_ctf_conserved(q, q_next, &[]) (debit = credit).
  9. q_next.state_root_t = sha256(b"turingosv4.challenge.accept.v1" ‖ prev_root ‖ canonical_hash(tx)).
```
