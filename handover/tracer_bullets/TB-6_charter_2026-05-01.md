## 0. Why this TB exists (one paragraph)

To TB-5, TuringOS has the kernel: two-channel ingress (`submit_agent_tx` / `emit_system_tx`), apply_one stage 1.5 pinned-pubkey verification, ChallengeResolve dispatch, formal RSP-1/RSP-2/RSP-3.0/RSP-3.1 admission spine, 5-holding CTF, 9-sub-field EconomicState, replay-deterministic chain. **All of this is exercised exclusively in `cargo test --workspace` via `InMemoryLedgerWriter`.** No production binary drives `Sequencer::apply_one`; `experiments/minif2f_v4/src/bin/evaluator.rs` does not import `turingosv4::state::sequencer`; `bus.rs:73` `sequencer: Option<Arc<Sequencer>>` is `None` in `main.rs`. No on-disk ChainTape from any LLM-driven run exists in TuringOS history. TB-6 closes that gap: at least one LLM-driven run traverses `rtool/context → Agent proposal → WorkTx → Sequencer::apply_one → LedgerWriter::commit → on-disk ChainTape → replay/verify_chain/reconstruct`. After TB-6, "smoke" and "tape" mean different things.

## 4. WP-canonical decision blocks (binding)

### 4.1 No new economic mutator in TB-6

TB-6 introduces **zero** new TypedTx variants, zero new TransitionError variants, zero new state-root domains, zero new monetary-invariant cascades. The kernel surface at TB-5 ship is sufficient to demonstrate ChainTape production wire-up. Any addition to economic state machinery is out of scope.

### 4.2 No agent chain-of-thought broadcast or persistence

Per architect ruling § 3.6 Atom 5 + the constitutional "selective shielding" axiom: the Agent audit trail records what the Agent **saw** (`prompt_context_hash` + `read_set`) + what it **submitted** (`proposal_cid` + `candidate_proof_cid` + `tx_id`) + how the system judged (`predicate_results`, `accepted_or_rejected`, `rejection_class`). It does NOT record private chain-of-thought / private model deliberation / private tool transcripts beyond what hits CAS as an artifact.

### 4.3 Smoke evidence renamed pre-TB-6; chain-backed evidence renamed "ChainTape smoke" / "smoke tape" only post-TB-6

Per architect ruling D5: the prior `handover/evidence/tb_{1..5}_smoke_*` directories are **smoke evidence** — paper trail, not chain. From TB-6 onward, when (and only when) an evidence directory contains a real on-disk ledger chain produced by a production binary, it may be called a **ChainTape smoke** / **smoke tape**. This charter holds itself to the new naming.

### 4.4 Audit mode = hybrid by risk class; TB-6 = production wire-up class

Per architect ruling D3: production wire-up requires Codex implementation audit + Gemini architecture audit. If Gemini at strategic tier is exhausted, ship-time evidence carries an explicit `degraded` label per TB-5 supplement precedent. **Self-audit alone is insufficient for TB-6 ship.** Every TB-6 atom that touches `experiments/minif2f_v4/src/bin/evaluator.rs` or `src/bus.rs` or `src/main.rs` needs an explicit dual-audit (or degraded with label) at ship gate.

### 4.5 cargo test --workspace remains canonical at every atom

Per architect ruling D4. Each atom commit body MUST report:
```
command = cargo test --workspace
workspace_count = N
failed = 0
ignored = M
```
Bare `cargo test` count is forbidden for TB-6 ship reporting.

### 4.6 No "RSP-M" / NodeMarket / Polymarket / CompleteSet work in TB-6

Per architect ruling § 4. The full RSP-M NodeMarket track (NodePosition, NodeMarketEntry, PriceIndex, CompleteSet, MarketOrder) is reserved-future post-TB-6 and is explicitly forbidden from this TB. WorkTx.stake stays a kernel admission gate, not a market position; ChallengeTx.stake stays a kernel challenge bond, not a Short/NO position; VerifyTx.bond stays a responsibility bond, not a market position. The future-form first-long economic interpretation is documented in the architect ruling but does NOT enter the codebase in TB-6.

### 4.7 STEP_B_PROTOCOL applies to bus.rs / main.rs touches

`src/bus.rs` and `src/state/sequencer.rs` are STEP_B-protected per CLAUDE.md + TB-2 P1-A. Any TB-6 atom that wires `Sequencer` into `TuringBus` or initializes a production `Git2LedgerWriter` from `main.rs` MUST use the parallel-branch A/B protocol per `STEP_B_PROTOCOL.md` and `feedback_step_b_protocol`. The evaluator binary `experiments/minif2f_v4/src/bin/evaluator.rs` is NOT STEP_B-protected (sub-crate experiment binary), but its imports of `turingosv4::state::*` cross the kernel boundary and warrant a smaller-scale phased commit.

## 8. Three declarative success proofs

TB-6 is provisionally green when all three are demonstrable end-to-end through the production wire-up path:

**Proof 1 — Production binary drives Sequencer to on-disk ChainTape**:
> An LLM-driven evaluator run on `mathd_algebra_107` (or equivalent simple problem) produces a `runtime_repo/` containing a real on-disk ledger chain. The chain has at least one `LedgerEntry` with valid `parent_ledger_root` + `resulting_ledger_root` + `tx_payload_cid` + `system_signature`. Tampering with any byte of any entry detectably breaks chain verification. (Atom 4 `verify_chaintape` exits 0 on the unmodified chain; non-zero on tampered chain.)

**Proof 2 — At least one accepted L4 + one rejected L4.E from production**:
> The production smoke produces at least one L4 entry (accepted WorkTx) and at least one L4.E entry (rejected WorkTx — natural or synthetic-with-explicit-label). The L4.E `RejectedSubmissionRecord` does NOT include `raw_diagnostic_cid` in its agent-facing serialized form (TB-1 P0-3 serde shield re-confirmed at production path). State / EconomicState reconstructable from L4 alone (L4.E is evidence-only, not state-bearing).

**Proof 3 — Agent audit trail records what mattered, not what's private**:
> Each Agent proposal in the run has an `AgentProposalRecord` (or equivalent) with `agent_id`, `prompt_context_hash`, `read_set`, `write_set`, `proposal_cid`, `candidate_proof_cid`, `tx_id`, `predicate_results`, `accepted_or_rejected`, `rejection_class`. None contain raw model deliberation transcripts, tool transcripts beyond CAS-stored artifacts, or any other form of chain-of-thought. The fork-visibility summary records `tx_count`, `failed_branch_count`, `rollback_count`, accepted/rejected tx_id sets.

After all three proofs are green:

> TuringOS production binary drives the runtime kernel. At least one LLM-driven run produces a walkable, replay-verifiable, signature-checked, CAS-linked on-disk ChainTape. The L4 / L4.E split is enforced at production path (not just in `cargo test`). The Agent audit trail is structurally bounded — it records what the Agent saw + submitted, not chain-of-thought. The kernel claims TB-1..TB-5 made are now verifiable from on-disk artifacts produced by an LLM-driven run, not only from `cargo test --workspace` in-memory state. The 5-TB ChainTape debt is closed. RSP-3.2 Slash, NodeMarket / RSP-M, and further economic surfaces become reachable in subsequent TBs without expanding the gap.
