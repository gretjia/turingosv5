## 0. One-line goal

Formalize RSP-1 admission as **two first-class TypedTx variants** on the canonical L4 — `TaskOpenTx` (task metadata registration) and `EscrowLockTx` (sole bounty funding path). **Delete** the TB-2 P0-B option (a) bridge at `src/state/sequencer.rs:205-215`. `WorkTx.stake` stays inline per WP § 14.1 + § 18 Invariant 5; **no `YesStakeTx` variant is introduced**. After TB-3, WorkTx admission is structural (looks up real ChainTape-recorded escrow + checks inline stake against agent solvency), not bridged from a synthetic `TxId(task_id.0)` lookup, and accepted WorkTx commits stake by debiting balances → stakes_t in a single conserved transition.

## 1. roadmap_exit_criteria_addressed

- **P3:3** — `work_tx must lock YES stake` — re-discharged through formal admission gate + accepted-WorkTx balance debit (stake commitment becomes a real economic-state mutation, not just a wire-field declaration).

## 3. WP-canonical decision blocks (must persist verbatim through any charter revision)

### 3.1 WP-canonical implementation shape

TB-3 follows the whitepaper-canonical implementation shape, NOT the literal Roadmap § 3 transaction list:

- `WorkTx.stake` (at `src/state/typed_tx.rs:232`) **remains inline**. It is NOT migrated to a `stake_lock_tx_id: TxId` reference.
- **No `YesStakeTx` TypedTx variant is introduced**. Adding one would be ABI-level work that contradicts the WP data-structure shape; doing it would also re-introduce the very same indirection class (synthetic ID matching) that the bridge at `sequencer.rs:205` was an apology for.
- The roadmap item `yes_stake_tx` (ROADMAP `handover/architect-insights/ROADMAP_9_PHASE_2026-04-29.md` § 3 P3 transactions list) is interpreted as the **semantic role of `WorkTx.stake`**, not a Rust ABI variant. Per WP § 17.6 layered ordering: ROADMAP § P3 is logical-tx narrative (叙事层); WP § 14.1 + § 5 Layer 4 + § 18 Inv 5 + § 19 RSP-1 + economic § 7 are implementation shape (实现层); on data-structure questions implementation shape wins.
- RSP-1 formal surface introduces only **two** new TypedTx variants:
  - `TaskOpenTx`
  - `EscrowLockTx`
- WorkTx admission requires (in order):
  1. `work.stake.micro_units() > 0` (stake declared)
  2. `task_markets_t[work.task_id].total_escrow.micro_units() > 0` (task funded)
  3. `balances_t[work.agent_id] >= work.stake` (solver solvent for the stake commitment)

WP authority cites:
- WP v2 § 14.1 Proposal Object — stake field is an inline structured value `{coin, yes, no}`.
- WP v2 § 5 Layer 4 Transition Ledger schema — `stake` is a first-class L4 row field (not a reference).
- WP economic § 7 — "Solver Agent 提交 work_tx 押注 YES_E" (solver bets via the work_tx itself).
- WP economic § 18 Invariant 5 — "YES/NO 是事件绑定的风险权利, 不是无抵押新币" (YES/NO are bound to the work_tx event).
- WP economic § 19 RSP-1 — "TaskMarket — 发布任务、广播价格、锁定奖金" (TaskMarket bundles task publishing with bounty locking).

### 3.2 No-double-counting decision

After TB-3 schema migration:

- **`EscrowsIndex` is the source of truth for escrowed money**. Each row records one accepted `EscrowLockTx`'s amount + depositor.
- **`TaskMarketEntry.total_escrow` is a derived aggregate / cached index**, NOT a money holding. It exists for O(1) WorkTx admission lookup; it must never act as an independent supply source.
- `monetary_invariant::total_supply_micro()` MUST count `Σ escrows_t[e].amount` and MUST NOT count `Σ task_markets_t[t].total_escrow`. Counting both would double-count every locked bounty by exactly its amount and effectively mint that amount each time `EscrowLockTx` runs.
- TB-1's `total_supply_micro` summed **six** holding subindexes (balances + escrows + stakes + claims + **bounty** + bond, see `src/economy/monetary_invariant.rs:90-94, :404-435`). TB-3 reduces this to **five** holdings (drop the `bounty` term entirely; bounty money has migrated to `escrows_t`).
- A new invariant test asserts the cache equals the source of truth:
  ```
  task_market.total_escrow == Σ escrows_t[e].amount where escrow.task_id == task_id
  ```
  This is the Art 0.2 守恒测试 contract for derived views (`assert_eq!(view, derive_from_tape(...))`); without it `total_escrow` becomes a parallel ledger.

### 3.3 TaskOpen / EscrowLock semantics

`TaskOpenTx` semantics:

- **Metadata-only**. Creates / registers a task market entry.
- Does NOT move money.
- Does NOT check sponsor balance (no money is debited).
- Does NOT make WorkTx admissible by itself — an opened-but-unfunded task carries `total_escrow == 0`, which fails the WorkTx admission gate § 3.1 step 2.
- Idempotency: reject with `TaskAlreadyOpen` if `task_markets_t.0.contains_key(&task_id)`.

`EscrowLockTx` semantics:

- The **only** RSP-1 bounty funding path. (No alternate code path inserts into `escrows_t` or grows `task_market.total_escrow`.)
- Checks `balances_t[sponsor] >= amount`; rejects with `InsufficientBalance` otherwise.
- Debits `balances_t[sponsor] -= amount`.
- Credits `escrows_t.0.insert(tx_id, EscrowEntry { amount, depositor: sponsor, task_id })`.
- Updates `task_markets_t[task_id].total_escrow += amount` and `.escrow_lock_tx_ids.insert(tx_id)` (cache update).
- Must pass `monetary_invariant` with the production-runtime empty exemption list `&[]` (debit = credit, conservation trivially holds).
- Pre-condition: `task_markets_t[task_id]` must EXIST (rejects with `TaskNotOpen` else); top-up to a non-existent task is the ghost-liquidity attack surface and is closed at admission.

### 3.4 WorkTx stake semantics — locked-on-accept (CHARTER DECISION, no longer open)

`WorkTx.stake` is the **event-bound YES stake commitment** for this WorkTx. The "event" is the WorkTx itself (per WP economic § 18 Inv 5). The semantic question raised by the user verdict — admission-only commitment vs locked-into-stakes_t-on-accept — is decided in this charter as **locked-on-accept**:

- Admission-time check (§ 3.1 step 3): `balances_t[work.agent_id] >= work.stake`. Insufficient balance → reject with `InsufficientBalance` to L4.E. No money moved.
- Acceptance-time mutation (NEW vs TB-2): the WorkTx arm of `dispatch_transition` produces a `q_next` with:
  - `balances_t[work.agent_id] -= work.stake`
  - `stakes_t.0.insert(work.tx_id, StakeEntry { amount: work.stake, staker: work.agent_id, task_id: work.task_id })`
- This lock IS an accepted state transition that must pass `assert_total_ctf_conserved(q, q_next, &[])` (debit = credit) and `assert_no_post_init_mint`. WP § 18 Inv 4 + Inv 5 jointly require this: stake = event-bound risk right; the event is the accepted WorkTx; the conversion happens at accept.
- Rejection semantics (per user verdict §14): a rejected WorkTx leaves `economic_state_t` UNCHANGED — no balance movement, no stakes_t insert. L4.E records evidence only; **L4.E never mutates economic state**.
- Slashing of a "bad" stake (e.g., predicate-failed work that nonetheless somehow committed money) is **not representable in TB-3**: rejected WorkTx has no economic mutation to slash. Future RSP-2/3 introduces explicit `ChallengeTx` + `ChallengeResolveTx` accepted state transitions to slash already-locked stakes (i.e., stakes that were locked because the WorkTx was accepted but later challenged successfully). That is the **only** slashing path TB-3 reserves; no L4.E-driven slash.
- YES/NO explicit split (per WP § 14.1 `stake = {coin, yes, no}`): in TB-3 minimum scope, `StakeEntry.amount: StakeMicroCoin` carries the locked coin amount only; explicit YES/NO sub-accounting deferred to RSP-2 (when `ChallengeTx` introduces a NO position). The current `StakeMicroCoin` newtype is sufficient for "locked stake awaiting outcome"; refining it into typed YES + NO positions is not in TB-3 scope.

This is the WP-canonical interpretation: stake is real money locked at accept; rejection records evidence only; future explicit accepted transitions resolve the lock. It is consistent with Law 2 ("Only Investment Costs Money"), § 18 Invariants 4 + 5 + 6 + 7, and economic § 7's "押注".

### 4.3 dispatch_transition arms — design

All arms remain pure (no I/O); apply_one writes the canonical L4 row.

```text
TaskOpen arm:
  1. parent_state_root match else StaleParent
  2. task_markets_t.0.contains_key(&task_id)? if yes → TaskAlreadyOpen
  3. q_next: insert TaskMarketEntry per § 4.1 (no money movement)
  4. assert_no_post_init_mint + assert_total_ctf_conserved(q, q_next, &[]) (trivial)
  5. q_next.state_root_t = sha256(b"turingosv4.task_open.accept.v1" ‖ prev_root ‖ canonical_hash(tx))

EscrowLock arm:
  1. parent_state_root match else StaleParent
  2. task_markets_t[task_id] EXISTS else TaskNotOpen
  3. balances_t[sponsor] >= amount else InsufficientBalance
  4. q_next:
       balances_t[sponsor] -= amount
       escrows_t.insert(tx_id, EscrowEntry { amount, depositor: sponsor, task_id })
       task_markets_t[task_id].total_escrow += amount
       task_markets_t[task_id].escrow_lock_tx_ids.insert(tx_id)
  5. assert_no_post_init_mint + assert_total_ctf_conserved(q, q_next, &[]) (debit = credit)
  6. q_next.state_root_t = sha256(b"turingosv4.escrow_lock.accept.v1" ‖ prev_root ‖ canonical_hash(tx))

WorkTx arm (refactor; bridge deletion + stake commitment):
  Steps 1-4 (parent-root, predicates, monetary-invariant ordering) inherited from TB-2.
  Step 5 (stake gate) — KEEP TB-2 inline check:
    if work.stake.micro_units() == 0 { return Err(StakeInsufficient); }
  Step 6 (escrow gate) — REPLACE bridge with structural lookup:
    OLD (sequencer.rs:197-215, deletion-target):
      let lookup_tx_id = TxId(work.task_id.0.clone());
      let has_escrow = q.economic_state_t.escrows_t.0.contains_key(&lookup_tx_id)
          || q.economic_state_t.task_markets_t.0.contains_key(&lookup_tx_id);
    NEW:
      let market = q.economic_state_t.task_markets_t.0.get(&work.task_id);
      let has_escrow = market.map_or(false, |m| m.total_escrow.micro_units() > 0);
      if !has_escrow { return Err(EscrowMissing); }
    The deletion-target comment at :205 is REMOVED with the bridge.
  Step 7 (NEW solvency gate):
    let bal = q.economic_state_t.balances_t.0.get(&work.agent_id)
        .copied().unwrap_or(MicroCoin::zero());
    if bal < work.stake.into() { return Err(InsufficientBalance); }
  Step 8 (NEW q_next economic mutation — accepted WorkTx commits stake):
    q_next.balances_t[work.agent_id] -= work.stake
    q_next.stakes_t.insert(work.tx_id, StakeEntry { amount: work.stake, staker: work.agent_id, task_id: work.task_id })
  Step 9 (existing): state_root advance via WORKTX_ACCEPT_DOMAIN_V1 (unchanged from TB-2).
  Step 10 (existing): assert_no_post_init_mint + assert_total_ctf_conserved(q, q_next, &[]).
    Unlike TB-2 (where economic_state_t was unchanged), this conservation check is now non-trivial: the
    debit-to-stakes invariant is the primary CTF check on the runtime spine.
```

### 4.4 New `TransitionError` variants

Three new variants in `src/state/typed_tx.rs::TransitionError` + matching `Display` arms:

- `TaskAlreadyOpen` — TaskOpen idempotency (task_id already in `task_markets_t`).
- `TaskNotOpen` — EscrowLock / WorkTx admission referenced a `task_id` not in `task_markets_t`.
- `InsufficientBalance` — EscrowLock sponsor / WorkTx solver lacks balance for the requested debit.

Existing variants reused: `EscrowMissing` (TB-2; now means "task exists but `total_escrow == 0`"), `StakeInsufficient` (pre-TB-2; for `work.stake == 0`), `StaleParent` (pre-TB-2), `MonetaryInvariantViolation` (TB-2).
