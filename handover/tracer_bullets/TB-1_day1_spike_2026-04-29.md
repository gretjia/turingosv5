## Implementation notes

1. **Hash function = `DefaultHasher` (16-char hex), not SHA-256, for Day-1.**
   Adding a `sha2` direct dep to `experiments/minif2f_v4/Cargo.toml` mutates
   the workspace `Cargo.lock`, which is itself in `genesis_payload.toml [trust_root]`.
   The constitution.md hash inside that manifest is sudo-protected (R-018), so
   any Cargo.lock churn during a TB-1-scope edit risks crossing the sudo line
   on a re-hash. Day-4 will upgrade to SHA-256 in a single commit alongside
   the h_vppu computation, where the Trust Root manifest is touched
   intentionally for both fields together. The DefaultHasher digest gives
   16 hex chars (64-bit) which is sufficient for retrieval-equivalence detection
   on a per-problem basis (only collisions across the same problem matter for
   step-4).
2. **Stamping happens at the run_oneshot prompt-build site only (Day 1 scope).**
   A `let stamp = |mut r: PputResult| -> PputResult { r.prompt_context_hash = Some(prompt_hash.clone()); r };`
   closure wraps each of the 4 `make_pput(...)` return sites in `run_oneshot`.
   `run_swarm` stays unstamped today; Day-2/3 wires it as ledger + per-tx
   hooks land. Until Day-2, swarm runs emit rows where `prompt_context_hash`
   is absent (None → skip-if-none), which is the explicit Day-1 contract.
3. **Trust Root manifest update is non-sudo per R-018 / F-2026-04-25-07.**
   Only the constitution.md SHA inside genesis_payload.toml requires sudo.
   The evaluator.rs entry was rehashed
   `c646853351...` → `7861c06abe10755c8cc08c750af20f8e650b24b5d9efc0169582b42a32369c6e`
   following R-014's protocol (recompute + update + run boot tests).
