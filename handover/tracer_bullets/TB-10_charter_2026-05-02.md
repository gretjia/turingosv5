## §0 Why TB-10 exists

**Architect spec verbatim** (line 1594 / Part C ruling 12+13):

```text
TB-10：Lean Proof Task Market MVP
目标：第一个可用产品：用户发任务，Agent 解题，系统验证，系统付款，dashboard 可审计。
必须：TaskOpenTx, EscrowLockTx, WorkTx, VerifyTx, FinalizeRewardTx, replay, dashboard
```

**Every primitive in the MUST list is already shipped** (TB-3..TB-8). TB-10 = thin user-facing wrapper that closes the loop:

```text
user opens Lean theorem statement      → TaskOpenTx           (TB-3 surface; new caller = user)
user locks bounty                       → EscrowLockTx         (TB-3 surface; new caller = user)
agent picks up open task                → reads task_markets_t (TB-3 surface; evaluator hook)
agent submits proof attempt              → WorkTx               (TB-3..TB-7R; signed by TB-9
                                                                  durable AgentId)
verifier confirms proof                  → VerifyTx + claim     (TB-4 + TB-8)
window closes / no challenge upheld      → FinalizeRewardTx     (TB-8 system-emitted)
solver balance increases                 → balances_t           (TB-3 + TB-8)
user views via dashboard / replay        → audit_dashboard +
                                            verify_chaintape   (TB-6 + TB-8)
```

**Net-new code surface** in TB-10:

- `src/bin/lean_market.rs` (new user-facing CLI binary; ~400 lines target)
- `src/runtime/bootstrap.rs` (new helper module; reusable preseed list factory exposing `Agent_user_0` alongside existing solver agents)
- `experiments/minif2f_v4/src/bin/evaluator.rs` user-task discovery hook (small extension)
- `src/bin/audit_dashboard.rs §11 User Tasks` view section
- `handover/evidence/tb_10_lean_market_mvp_smoke_2026-05-02/` (real-LLM end-to-end smoke evidence)

**NOT in scope** (architect-listed forbidden boundary):

- ❌ NEW typed_tx variant
- ❌ user-callable system-emitting surface (e.g. `lean_market settle`) — Anti-Oreo violation
- ❌ post-init mint via user budget — Agent_user_0 funds at on_init only
- ❌ NodeMarket / NodePosition / CompleteSet / MarketSeed (TB-11+)
- ❌ arbitrary-Lean source ingest (defer to TB-13 Beta)
- ❌ user-driven escrow refund / task expiry (defer to TB-12 RSP-3.2 + TB-13)
- ❌ multi-org / cross-host keystore federation

## §1 One-line goal

```text
Goal: A human user runs `lean_market post-task --problem mathd_algebra_171 --bounty 100000`
      → ChainTape gains 2 L4 entries (TaskOpen + EscrowLock) signed by Agent_user_0.
      Then runs `evaluator --task-mode user` → real-LLM solver picks up the open task,
      produces WorkTx → VerifyTx → FinalizeReward → solver balance += 100_000 micro,
      sponsor balance -= 100_000 micro.
      Then runs `lean_market view` + `audit_dashboard` + `verify_chaintape` → all 3 views
      converge: same task_id, same payout, same chain root, replay self-contained.
```

The user's solver-payout view IS the canonical economic view (`EconomicState.balances_t`) — there is no parallel f64 ledger, no cross-problem sidecar, no shadow state. TB-9's Wallet projection collapse is load-bearing here.

### Atom 1 — Runtime preseed factory + Agent_user_0 budget (Class 2)

**Action**: introduce `src/runtime/bootstrap.rs` exposing `default_pput_preseed_pairs() -> Vec<(AgentId, MicroCoin)>`. Wire evaluator's existing preseed block to call this factory instead of the inline literal. Factory returns:

```rust
pub fn default_pput_preseed_pairs() -> Vec<(AgentId, MicroCoin)> {
    let mut pairs = vec![
        // TB-7.7 sponsor (existing role; preserved for evaluator self-funded mode)
        (AgentId("tb7-7-sponsor".into()), MicroCoin::from_micro_units(10_000_000)),
        // TB-10 user sponsor (new; lean_market CLI signs as this agent)
        (AgentId("Agent_user_0".into()),  MicroCoin::from_micro_units(10_000_000)),
    ];
    for i in 0..10 {
        pairs.push((AgentId(format!("Agent_{i}")), MicroCoin::from_micro_units(1_000_000)));
    }
    pairs
}
```

`Agent_user_0` gets the same 10_000_000 micro (~10 Coin) sponsor budget as `tb7-7-sponsor` — plenty for ~100 EscrowLock at 100_000 each.

**Why factory not literal**: lean_market CLI must bootstrap chaintape with the SAME preseed list (so when the user runs lean_market first, then evaluator second, both see the same initial QState). Factory ensures consistency and is the single source of truth.

**Constitutional gate**: this is on_init (genesis QState construction), not post-init mint. `assert_no_post_init_mint` fires on every subsequent typed_tx but the genesis QState itself is the substrate, not a mutator. Identical pattern to existing `tb7-7-sponsor` preseed at `evaluator.rs:716-731`.

**Tests**:
- U1: `default_pput_preseed_pairs` returns 12 entries (1 tb7-7-sponsor + 1 Agent_user_0 + 10 Agent_i).
- U2: each entry has positive balance (no zero-funded agent).
- U3: `genesis_with_balances(&default_pput_preseed_pairs())` produces a QState whose balances_t Σ matches expected sum.
- I1: chaintape bootstrapped via this factory accepts a TaskOpen + EscrowLock signed by Agent_user_0 with bounty=100_000 (smoke-style integration; uses ZERO sig like existing pattern).
