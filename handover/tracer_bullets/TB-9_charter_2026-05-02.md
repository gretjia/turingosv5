## §0 Why TB-9 exists

TB-7 / TB-7R wired per-LLM-proposal Ed25519 signatures into the canonical L4 path. **The keypair lifecycle is run-local** (process memory, `runtime_repo/agent_pubkeys.json` written per run, fail-closed-on-existing). Architect ruling 2026-05-01 D2 explicitly forbade calling this "durable identity" — it is *run-local identity, not durable reputation identity*.

This caveat is **not free**. Three concrete consequences are inherited by every downstream TB unless TB-9 closes them:

1. **TB-10 Lean Proof Task Market MVP** wants payout addresses that survive across runs; without durable identity, every reboot mints fresh AgentIds, so balance accumulation is impossible.
2. **TB-11 NodePosition / PriceIndex (no trading)** aggregates positions per AgentId; run-local AgentIds yield market signal without owner.
3. **`verify_chaintape` cross-run replay** currently rebuilds AgentId → pubkey from a per-run manifest file. A reviewer who only has the L4 ChainTape cannot independently verify a WorkTx signature without the manifest sidecar — i.e., **the canonical path is not yet self-contained**.

Concurrently, the runtime carries a parallel f64 ledger in `WalletTool` (`src/sdk/tools/wallet.rs:24`) that pre-dates `EconomicState.balances_t` (introduced CO P2.0). Every dispatch arm since TB-3 has mutated `EconomicState.balances_t` (MicroCoin); the legacy `WalletTool.balances` (f64) is mutated only by `bus.rs` market shares code (the v3 simulation path that ChainTape does not touch). **Two ledgers, one canonical chain** — this is the literal definition of `feedback_proxy_running_vs_source_drift`-style trap waiting for the next reviewer to step on it.

Per architect directive 2026-05-02 ruling 13 + LATEST §9: TB-9 closes both gaps in one bundle. Durable AgentRegistry **and** Wallet Projection are co-located because they share the same invariant — one canonical record, replayable from L4, no shadow ledger.

The four 2026-05-02 decision records (POLYMARKET_CORE / CPMM_MINT_AND_SWAP / MARKET_SEED_NO_GHOST_LIQUIDITY / LAMARCKIAN_AUTOPSY_BOLTZMANN) remain forward-binding on TB-9: no ghost identity (mirror of no ghost liquidity), no agent-submitted system tx, no predicate override at the registry layer.

## §1 One-line goal

```text
Goal A — Durable AgentRegistry:
  Every AgentId observed at WorkTx-signing time is registered exactly once on canonical L4
  via a system-emitted AgentRegisterTx whose payload binds (agent_id, public_key,
  registered_at_logical_t). The registry survives across evaluator reboots via an encrypted
  keystore at `~/.turingos/keystore/agent_keystore.enc` (mirror of system_keypair pattern).
  verify_chaintape rebuilds q.agent_registry_t from L4 alone (no per-run manifest sidecar).

Goal B — Wallet Projection:
  WalletTool retains its TuringTool surface but holds zero owned f64 state. balance(agent)
  reads `EconomicState.balances_t[AgentId(agent)]` via an injected reference. The legacy
  bus.rs market f64 mutator path (debit_wallet / credit_wallet / record_shares — all
  pre-ChainTape v3 simulation code) is deleted. EconomicState.balances_t is the
  single canonical source of agent balance.
```

The conservation invariant `Σ balances_t + Σ escrows_t = total_supply_micro` is preserved exactly across the WalletTool collapse — Goal B is a **deletion of a parallel ledger that was never load-bearing on the chain path**, not a migration of state. There is no f64 ↔ MicroCoin transfer step.

## §7 Open ratification questions (for Atom 0.5)

These are **NOT shipping blockers** — they are charter ratification points to resolve at TB-9 Atom 0.5 before Atom 1 begins. Per `feedback_no_fake_menus`: each question carries a **proposed default**; Atom 0.5 ratification document states each as the recommendation, asks user to RATIFY or override.

### §7.1 Keystore file shape

**Recommendation**: **single encrypted file** at `~/.turingos/keystore/agent_keystore.enc` containing serialized `BTreeMap<AgentId, AgentSecretBytes>`, mirroring `system_keypair_v{epoch}.enc` precedent exactly. **Reason**: one file = one fsync = one atomic rename; mirrors the existing keystore precedent (`src/bottom_white/ledger/system_keypair.rs:417-463`); no dir-walk for multi-agent runs; KDF/nonce reuse on the encrypted-at-rest layer.

**Override candidate**: per-agent file (`~/.turingos/keystore/agents/<agent_id>.key`). Only justified if multi-org concurrent-write hits the keystore — out of scope until TB-16+.

### §7.2 `q.agent_registry_t` placement

**Recommendation**: **top-level QState field**, NOT under `EconomicState`. **Reason**: pubkey is identity, not economic state; preserves the `economic_state_has_nine_sub_fields` test at `q_state.rs:570` (currently a structural invariant); mirrors how `head_t / temporal_t / agent_audit_trail_t` live at QState top level, not under EconomicState.

**Override candidate**: extend `EconomicState` to 10 sub-fields and update the test. Only justified if downstream economic invariants (e.g. reputation-weighted price) require coupling — not the case for TB-9.

### §7.3 AgentRegister emission trigger

**Recommendation**: **first verified WorkTx-with-new-pubkey** (evaluator pre-WorkTx-submit hook checks `q.agent_registry_t`, emits AgentRegister on absence, awaits state_root advance, then submits WorkTx). **Reason**: ties identity to actual on-chain participation (no registration of agents who never propose); deterministic by replay (every L4 WorkTx is preceded by a corresponding AgentRegister); mirrors TB-8 sequencing pattern (`tb8_emit_finalize_after_verify` + `tb8_await_state_root_advance`).

**Override candidate (a)**: explicit boot-time emit (every keystore agent gets an AgentRegister at evaluator start, regardless of whether they ever propose). Justified if evaluator boot must publish a stable agent set for downstream NodeMarket — not yet (NodeMarket = TB-11+).

**Override candidate (b)**: register-on-first-sign (intercept `AgentKeypairRegistry::sign`). Coupling auth-primitive with chain-mutation is undesirable; rejected.

### §7.4 WalletTool collapse depth

**Recommendation**: **read-only projection wrapper retained** as the TuringTool surface; all f64 owned-state deleted; mutators deleted; bus market path deleted. **Reason**: minimum-blast-radius — keeps the TuringTool trait registration intact for tool-discovery code (`src/bottom_white/tools/registry.rs`); removes parallel ledger; explicit `balance(agent, &EconomicState)` projection signature documents the coupling.

**Override candidate**: full deletion of `WalletTool` itself (remove from tool registry). Justified if no downstream caller needs `wallet.balance()` API surface — but `bus.balance()` and audit dashboard read the projection, so the wrapper is load-bearing.

### §7.5 Legacy bus market path fate

**Recommendation**: **delete** the bus.rs market f64 mutator code (lines ~287-490) entirely along with its tests. **Reason**: this is v3 simulation pre-dating ChainTape (TB-3 onward); zero production binary calls these paths; per `feedback_chaintape_wire_up_priority`, ChainTape is the canonical path; legacy code is debt awaiting amplification.

**Override candidate**: `#[deprecated]` annotation pointing to typed_tx. Justified if any downstream non-binary consumer (research notebooks, V3-compat scripts) still uses the API. None known. Default to delete; if Atom 0.5 surfaces a user, downgrade to deprecate.
