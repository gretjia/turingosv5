## §0 Why this TB exists (one paragraph)

TB-6 closed Frame A — `production binary triggers Sequencer::apply_one + on-disk LedgerEntry chain + replay verifier`. But the **real LLM activity** (proposal generation, Lean verification, OMEGA accept) still routes through the legacy `bus.append` / `bus.append_oracle_accepted` path, NOT through `bus.submit_typed_tx → Sequencer::apply_one`. The `handover/evidence/tb_6_chaintape_smoke_2026-05-01/` chain contains exactly 2 entries (1 synthetic TaskOpen + 1 synthetic zero-stake WorkTx) — none of the 20 LLM proposals on `mathd_algebra_107` traverse the chain. PputResult is computed by the evaluator's in-memory accumulator and is NOT chain-derivable. TB-7 closes Frame B: every meaningful LLM action becomes an on-chain `WorkTx`/`VerifyTx` so PputResult is verifiable from L4 + L4.E + CAS alone, not from evaluator stdout.

## §1 One-line goal

Every meaningful LLM proposal in `evaluator::run_swarm` (append branch + complete/OMEGA branch) routes through `bus.submit_typed_tx` as the **AUTHORITATIVE** state-mutation path (legacy `bus.append` either removed, projected from ChainTape, or shadow-only) and lands in L4 (accepted) or L4.E (rejected) on the production chaintape, with `ChainDerivedRunFacts` (NOT full PputResult) bit-exact-equal to the evaluator's structural facts on a defined field set. Frame B closure is achieved iff Gates 1–7 (§8) are GREEN.

## §4 WP-canonical decision blocks (binding once ratified)

### 4.0 Authoritative path requirement (architect ruling D1; load-bearing)

**`bus.submit_typed_tx` is the AUTHORITATIVE proposal-write path. `bus.append` and `bus.append_oracle_accepted` MUST become one of:**

1. **Removed** from authoritative state-mutation surface (deleted from evaluator hot path);
2. **Read-only derived projection** from ChainTape (e.g., a view materialized by replay from L4);
3. **Explicitly marked `// shadow_only:` at every retained call site** — in which case TB-7 §0 / §1 / §8 MUST NOT claim Frame B closure (the charter degrades to "Frame B-shadow-only" and a follow-up TB is required to close).

**"Also emit" framing is forbidden.** Adding a parallel `bus.submit_typed_tx` call alongside an unmarked `bus.append` does NOT satisfy this clause; legacy append must be option (1), (2), or (3) as listed.

Rationale: only options (1) or (2) discharge kill criterion **P1:1 — Agent cannot bypass `submit_typed_tx` / wtool to mutate state** under real LLM activity. Option (3) is a documented degradation, not a closure. See ruling §3.1 / §4 Gate 1 / §4 Gate 7.

### 4.1 No new TypedTx variant

Per WP § 14.1: existing `WorkTx` + `VerifyTx` cover proposal + verification. TB-7 wires evaluator → these existing variants via the §4.0 authoritative path; does NOT introduce new variants. Charter §6 #6 rule from TB-6 inherited.

### 4.2 Agent keypair = per-agent runtime-generated Ed25519 + AgentPublicKeyRegistry

Each `agent_id` (e.g., `n1`, `swarm_a`, `swarm_b`) gets a runtime-generated Ed25519 keypair at evaluator startup. The public key is recorded in a new on-disk manifest `<runtime_repo>/agent_pubkeys.json` (analogous to TB-6's `pinned_pubkeys.json` but agent-side). Private key lives in process memory only and is dropped at evaluator exit. **Structurally analogous to TB-5's PinnedSystemPubkeys but for agents** — same proven shape.

WorkTx.signature is now a real Ed25519 signature over the canonical WorkTx digest. Replay verifier (extended Atom 4 path) re-verifies every WorkTx.signature against the agent_pubkeys manifest.

**Caveat (architect ruling D2)**: This is **run-local identity, not durable reputation identity.** The keypair lifecycle is process-bound; private key dropped at evaluator exit; `agent_pubkeys.json` resides in the run's `runtime_repo/`. Cross-run reputation, NodeMarket identity, or long-term agent economic identity REQUIRE a separate TB (`Persistent AgentRegistry + agent keystore`) — explicitly NOT covered here. TB-7 MUST NOT claim "Agent identity is on-chain" without this caveat.

### 4.3 OMEGA-accept path scope (NARROWED)

The full OMEGA → settlement loop is `WorkTx → VerifyTx → ChallengeWindow → FinalizeRewardTx`. TB-7 covers ONLY the front half:
- Accepted OMEGA proposal → `WorkTx` with non-zero stake + signature
- Lean verification result → `VerifyTx` with bond + verdict
- ChallengeWindow stays OPEN at TB-7 ship (no slash; no settlement)
- `FinalizeRewardTx` → DEFERRED to TB-11 RSP-4 (SettlementEngine + ContributionDAG)
- `SlashTx` → DEFERRED to TB-9 RSP-3.2

Rationale: settlement requires reward distribution math (creator royalty + verifier fee + reuse-DAG flow) which is RSP-4 territory. Architect ruling 2026-05-01 § 4.5 sequences this as TB-11. TB-7 stops at "the chain shows who proposed what + who verified what + which were accepted/rejected" without claiming the economics are settled.

### 4.4 Chain-derived run facts (NOT full PputResult) — bit-exact on structural field set (architect ruling D4)

Atom 5 introduces `runtime::chain_derived_run_facts::compute_run_facts_from_chain(runtime_repo, cas) -> ChainDerivedRunFacts`.

**Bit-exact field set** (verifiable from L4 + L4.E + CAS alone):

```
solved
verified
tx_count
proposal_count
golden_path_token_count
gp_payload
gp_path
gp_proof_file
tactic_diversity
tool_dist
failed_branch_count
```

**Excluded fields** (NOT chain-derivable; remain as evaluator stdout / in-memory only):

```
total_wall_time_ms
verifier_wait_ms
pput_runtime
pput_verified
pput_m_verified
h_vppu
```

**Pre-condition for `golden_path_token_count` chain-derivation**: token counts MUST be persisted as `ProposalTelemetry` CAS objects (see §4.5 / §5.1 Atom 1.5). If token counts are NOT in CAS for any reason, `golden_path_token_count` MUST be removed from the bit-exact set — the chain cannot derive what the chain does not record. Calling Atom 5 output "chain-derived PPUT" without token_counts in CAS is forbidden (§6 #32).

If divergence on any included field: the in-memory accumulator is treated as **wrong** (the chain is canonical post-TB-7). Tests force any drift to surface as an Atom 5 test failure.

**Naming**: prior draft language ("chain-derived PPUT") is retired. Use `ChainDerivedRunFacts`. README Q10 reads: "Does chain-derived **run facts** match in-memory structural facts?"

### 4.5 ProposalTelemetry CAS object (architect ruling D5; new Atom 1.5)

Every WorkTx written through `bus.submit_typed_tx` carries `proposal_cid` pointing to a CAS-resident `ProposalTelemetry` object with the following shape:

```json
{
  "agent_id": "<string>",
  "prompt_context_hash": "<hex>",
  "proposal_artifact_cid": "<cid>",
  "candidate_tactic": "<string>",
  "token_counts": {
    "prompt_tokens": 0,
    "completion_tokens": 0,
    "tool_tokens": 0
  },
  "tool_calls": [],
  "branch_id": "<string>",
  "parent_tx": "<TxId or null>"
}
```

Without this object, §4.4 cannot claim `golden_path_token_count` as chain-derived. Without it, Gate 5 (CAS proposal telemetry retrievability) fails. New module: `src/runtime/proposal_telemetry.rs`. New Atom 1.5 in §7.
