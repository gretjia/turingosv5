## 0. v1 → v2 redesign banner

Charter v1 (`1b60237`) proposed `ChallengeResolveTx` as a system-emitted economic mutator but **failed to specify a live ingress / signature gate**. Codex round-1 audit B2 verified the gap: `Sequencer::submit` accepts bare `TypedTx` from any caller (`src/state/sequencer.rs:863`), and `dispatch_transition` does not verify `system_signature` (entire src/ has zero signature-verification calls; `PinnedSystemPubkeys` is replay-side only). Result: any agent could craft `ChallengeResolveTx{Released}` and refund any challenge bond at any time. **Constitutional-level VETO** (Anti-Oreo: agent ≠ direct state writer).

v2 redesign per architect directive § 4 Q2 + § 5: **two-channel ingress architecture**.

```text
Sequencer::submit_agent_tx(...)   — agent variants only
Sequencer::emit_system_tx(...)    — system-emitted variants only; system_signature live-verified
```

**Renamed + restructured**: TB-5 v2 is "System-Emitted Resolution Gate + Challenge Bond Release," NOT "Challenge Resolve." The substrate barrier (TB-5.0) precedes any resolution semantics (TB-5.1).

### 4.2 Two-channel ingress architecture (Option 1; binding per directive § 4 Q2)

```text
Sequencer::submit_agent_tx(envelope: AgentSubmissionEnvelope)
  Accepts ONLY agent-submitted variants:
    WorkTx | VerifyTx | ChallengeTx | TaskOpenTx | EscrowLockTx | ReuseTx
  Returns SystemTxForbiddenOnAgentIngress for system variants:
    ChallengeResolveTx | FinalizeRewardTx | TaskExpireTx | TerminalSummaryTx

Sequencer::emit_system_tx(command: SystemEmitCommand)
  Internal-only entry path for system-emitted variants.
  Constructs the typed tx + signs with system_keypair internally.
  Verifies system_signature via PinnedSystemPubkeys at dispatch entry.
  Returns InvalidSystemSignatureLive on signature mismatch.
```

**Critical structural property** (charter § 11.4 binding):

> `system_signature` is NOT a schema-only field. It MUST be live-verified at the ingress gate, OR constructed inside `emit_system_tx` from the system keypair. There is NO code path where a `ChallengeResolveTx` reaches `dispatch_transition` without its `system_signature` being fresh-verified or freshly-constructed.

Existing `Sequencer::submit()` (the bare-TypedTx path at `src/state/sequencer.rs:863`) is **either** narrowed to delegate to `submit_agent_tx` (rejecting system variants pre-dispatch) **or** removed entirely. Charter v2 implementation Atom 2 picks the safer option (narrow + reject); Atom 3 may remove the legacy alias once all internal callers migrate.

### 4.3 system_signature is live-verified at ingress gate

Per architect directive § 11.4:

> `system_signature` 不能只是 schema 上的字段。它必须在 live ingress / dispatch 中被验证，或者由不可伪造的 internal `emit_system_tx` 构造。否则它就是"签名形状的注释"。

Implementation: `emit_system_tx` constructs the system_signature internally using the runtime's `Ed25519Keypair` (already in `Sequencer.keypair: Arc<Ed25519Keypair>`). Dispatch verifies via `PinnedSystemPubkeys::verify` (currently used at replay-side per `src/bottom_white/ledger/system_keypair.rs:505+524`; TB-5.0 promotes this verification to live dispatch entry).

CI test (TB-5.0): `emit_system_tx_rejects_missing_or_bad_signature` — submitting a `ChallengeResolveTx` with `system_signature: SystemSignature::from_bytes([0u8; 64])` (or any invalid signature) through `emit_system_tx` produces `InvalidSystemSignatureLive` and zero state mutation.

### 4.4 9-sub-field EconomicState invariant + 5-holding CTF preserved

Per architect directive § 7 Q1 + Q2:

- `challenge_window_length` may be added to `TaskMarketEntry` (entry-shape additive; default 10) for **future TB-6 use** but TB-5.1 does NOT use it for closure math.
- `accepted_at_round` is **NOT added** to `StakeEntry` in TB-5 (defer per directive Q2 — "不要为了 TB-6 预先污染 TB-5 schema"). TB-6 adds it when window-closure math is actually installed.
- `ChallengeCase.status: ChallengeStatus` is added (entry-shape additive serde-default; default = `Open`). Used by Released and UpheldDeferred to mark resolution outcome **without removing the entry** (per directive § 7 Q6 + § 11.3).

The 9 sub-fields of `EconomicState` are unchanged: balances + escrows + stakes + claims + reputations + task_markets + royalty_graph + challenge_cases + price_index.

Conservation count remains **5 holdings**: balances + escrows + stakes + claims + challenge_cases.bond. Released sets `bond = 0` on the resolved entry (case stays in challenge_cases_t with status=Released; bond contribution to CTF sum becomes 0; delta moves to balances).

### 4.5 ChallengeResolveTx schema (system-only)

```rust
/// TRACE_MATRIX TB-5 charter v2 § 4.5 — system-emitted challenge resolution.
/// Cannot enter Q via agent ingress; must come through Sequencer::emit_system_tx.
/// Released refunds challenger bond + sets ChallengeCase.status = Released.
/// UpheldDeferred sets ChallengeCase.status = UpheldDeferred; ChallengeCase
/// preserved (not removed) for future TB-6 slash routing.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize, Default)]
pub struct ChallengeResolveTx {
    pub tx_id: TxId,                                //  1
    pub parent_state_root: Hash,                    //  2
    pub target_challenge_tx_id: TxId,               //  3 — keys challenge_cases_t lookup
    pub resolution: ChallengeResolution,            //  4
    pub epoch: SystemEpoch,                         //  5
    pub timestamp_logical: u64,                     //  6

/// TRACE_MATRIX TB-5 charter v2 — resolution outcome enum (per directive § 7 Q4).

#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash, Serialize, Deserialize)]

pub enum ChallengeResolution {
    Released = 0,
    UpheldDeferred = 1,

impl Default for ChallengeResolution { fn default() -> Self { Self::Released } }
```

### 4.6 Released path semantics (TB-5.1 active)

```text
case = q.economic_state_t.challenge_cases_t.0.get(&target_challenge_tx_id)
       (verified at admission step 2 below)

Admission steps:
  1. parent_state_root match else StaleParent.
  2. case exists else ChallengeNotFound (NEW TransitionError).
  3. case.status == Open else AlreadyResolved (NEW TransitionError).
     (Per directive § 7 Q6: Released cannot run twice; status-flip is
      the idempotency guard.)
  4. resolution == Released (this arm).

q_next mutations on Released:
  balances_t[case.challenger] += case.bond       // refund
  case.bond = MicroCoin::zero()                  // zero out (CTF still sums it as 0)
  case.status = ChallengeStatus::Released        // audit trail preserved

Invariants:
  assert_no_post_init_mint(tx, q)
  assert_total_ctf_conserved(q.economic_state_t, q_next.economic_state_t, &[])
    → debit challenge_cases.bond (Σ-X) = credit balances (Σ+X); CTF closes.

state_root advance:
  q_next.state_root_t = sha256(CHALLENGE_RESOLVE_DOMAIN_V1 ‖ prev ‖ canonical_hash(tx))
```

**Per directive § 11.2**: "Released ≠ settlement." Released only refunds challenger bond. It does **NOT**: confirm Solver guilt-free / confirm Verifier responsibility-free / finalize task / pay out reward.

### 4.7 UpheldDeferred path semantics (TB-5.1 marker only)

```text
Admission steps 1-3: same as Released.
  4. resolution == UpheldDeferred (this arm).

q_next mutations on UpheldDeferred:
  case.status = ChallengeStatus::UpheldDeferred  // marker only
  // bond stays > 0 (preserved for TB-6 slash routing)
  // challenger / opened_at_round / target_work_tx all unchanged
  // balances_t unchanged
  // stakes_t unchanged

Invariants:
  assert_total_ctf_conserved(...) trivially holds — economic_state_t
  is bit-identical except for the ChallengeStatus field flip (which is
  not in any of the 5 holding sums; CTF is invariant under status changes).
```

**Per directive § 11.3**: "UpheldDeferred ≠ slash." UpheldDeferred does NOT decrement Solver YES stake, Verifier bond, or any other agent's holding. Slash is RSP-3.2 / TB-6.

### 4.8 No state mutation on Verifier bond / Solver YES stake (RSP-3.x boundary)

Per architect directive § 5.4 + § 11.2:

- TB-5.1 Released: refund challenger bond ONLY. Solver `stakes_t[work.tx_id]` and Verifier `stakes_t[verify.tx_id]` are **untouched**.
- TB-5.1 UpheldDeferred: marker only. Same — Solver + Verifier stakes untouched.
- Solver / Verifier release (success path) is RSP-4 FinalizeRewardTx territory.
- Solver / Verifier slash (failure path) is RSP-3.2 / TB-6 SlashTx territory.

### 4.9 SystemTxForbiddenOnAgentIngress error class (TB-5.0 core)

Per architect directive § 7 Q3:

> 命名应更直接: `SystemTxForbiddenOnAgentIngress` 或 `SystemVariantRejectedAtAgentSubmit`.
>
> 这是 TB-5.0 的核心 error，不是 over-engineering。

New TransitionError variants in TB-5:
- `SystemTxForbiddenOnAgentIngress` — agent submitted ChallengeResolveTx / FinalizeRewardTx / TaskExpireTx / TerminalSummaryTx through `submit_agent_tx`. Maps to `L4ERejectionClass::PolicyViolation`.
- `ChallengeNotFound` — `target_challenge_tx_id` not in `challenge_cases_t.0`. Maps to `PolicyViolation`.
- `AlreadyResolved` — `case.status != Open`. Maps to `PolicyViolation`.
- `InvalidSystemSignatureLive` — system_signature live-verification failed at `emit_system_tx` ingress. Maps to `PolicyViolation`. Distinct from TB-3's `InvalidSystemSignature` (which was reserved-but-unused since CO1.1.4-pre1 — TB-5 actually emits this class).

### 4.10 No window-closure / round-tick / counterexample-evaluation

Per architect directive § 5.4 + § 7 Q1 + § 11.1:

- TB-5 does **NOT** compute `q.q_t.current_round - opened_at_round >= challenge_window_length`. That is RSP-3.2 / TB-6.
- TB-5 does **NOT** advance `q.q_t.current_round` from any dispatch arm.
- TB-5 does **NOT** evaluate `counterexample_cid` against the predicate registry to decide Released vs UpheldDeferred. The resolution decision is **out-of-band** (operator-driven; tests parameterize). Per directive § 11.1: "ChallengeResolveTx 不是'系统判断 counterexample 真伪'的主观裁判. 它只是 system-emitted resolution signal. 不要让 LLM 或 Agent 当 judge."
