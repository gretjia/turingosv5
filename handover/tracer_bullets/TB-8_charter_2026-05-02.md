## §0 Why TB-8 exists

TB-7R closed Frame B authoritative path: every externalized LLM proposal lands in `L4` (accepted) or `L4.E` (rejected) with predicate evidence resolvable from CAS. The current 5-step compile loop is therefore:

```text
Proposal → Ground-Truth Feedback → [STUBBED settlement] → Logging → Capability Compilation → ↑H-VPPUT
                                          ↑
                                          TB-8 closes this node
```

`TypedTx::FinalizeReward(_)` dispatch arm at `src/state/sequencer.rs:622` returns `TransitionError::NotYetImplemented`. `SystemEmitCommand::FinalizeReward { ... }` is explicitly deferred at `sequencer.rs:928` (`"Future RSP-3.2 / RSP-4 additions (NOT in TB-5 scope)"`). `ClaimsIndex` is a stub never written by any dispatch arm.

TB-8 wires the **minimal** version of this: single-solver, single-verifier, no royalty, no DAG, no NodeMarket, no AMM. Just enough to (a) close the compile loop at the settlement node, and (b) provide the trust primitive that lets a Lean Proof Task Market MVP (TB-10) be built (per `feedback_launch_priority` + directive ruling 12).

Per architect directive 2026-05-02 ruling 13: **NodeMarket starts after durable identity AND Lean Proof Task Market MVP** — i.e., not until TB-11 at earliest. Slash (originally roadmap RSP-3.2 / TB-9) is re-sequenced to post-payout-and-evidence territory. Payout *is* the invariant; slash hardens it.

The four 2026-05-02 decision records (POLYMARKET_CORE / CPMM_MINT_AND_SWAP / MARKET_SEED_NO_GHOST_LIQUIDITY / LAMARCKIAN_AUTOPSY_BOLTZMANN) are **forward decisions** — they govern TB-11..TB-15. TB-8 explicitly does not invoke them but is bound by their hard prohibitions (no ghost liquidity, predicate-first truth, no agent-submitted system tx).

## §1 One-line goal

For every L4 `WorkTx` accepted via OMEGA path with a closed `ChallengeWindow` and no upheld challenge, the system emits exactly one `FinalizeRewardTx` that atomically:

```text
debit  escrows_t[lock_tx_id].amount  -= reward
credit balances_t[solver]            += reward
mark   claims_t[claim_id].status      = Finalized
emit   L4 LedgerEntry(FinalizeRewardTx) signed by runtime keypair
```

…with the conservation invariant `Σ balances_t + Σ escrows_t = total_supply_micro` preserved exactly across the transition, and `chain_oracle_verified=true` runs producing replayable evidence that `accepted L4 WorkTx → L4 FinalizeRewardTx` happened on chain (not in stdout).

## §7 Open questions for architect ratification (Atom 0.5)

1. **`ClaimEntry` schema extension** (Atom 1): TB-8 proposes adding `task_id` / `escrow_lock_tx_id` / `work_tx_id` / `verify_tx_id` / `status` / `challenge_window_close_logical_t`. Does the architect prefer a minimal `{ amount, claimant, status, lookup_refs }` packed shape, or are these 6 fields ratified verbatim?

2. **Idempotency error variant naming** (Atom 3): existing `TransitionError::ClaimAlreadySlashed` reads "slashed-only" but the more general "already terminal-resolved" condition needs surfacing. Add `ClaimAlreadyFinalized`, or reuse `ClaimAlreadySlashed` semantically broadened, or introduce `ClaimAlreadyResolved(ClaimStatus)`?

3. **Zero-window MVP (Atom 4) vs. minimum-1-block window**: solo-run zero-window is the literal `feedback_launch_priority` minimal-payout. But `Art.III.4 challenge_window_closed` predicate semantically requires a window. Does the architect ratify zero-window (window collapses to zero in solo mode) or require ≥1 logical-tick gap?

4. **Conservation invariant implementation**: `debug_assert` only (release-mode trusts the tested-in-debug path) vs. `assert` (release-panic if invariant violated). TB-8 default: `debug_assert` for the in-arm check + dedicated `cargo test --release` unit test that validates invariant. Acceptable?

5. **`reward_factor`** (Atom 1): TB-8 picks `claim.amount = task_market_entry.total_escrow` for single-solver. If a task has total_escrow=100 and one solver, solver gets 100. No platform fee, no reserve. Ratify, or reserve a platform-fee placeholder field for a later TB?

These are NOT shipping blockers — they're charter ratification points. TB-8 Atom 0.5 produces the architect ruling document, mirrored on TB-7R 0.5 precedent.
