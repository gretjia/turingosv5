## 0. Frame

The Phase-7 grill ships 8 static questions to an LLM that only writes the synthesis. The architect wants it true Software-3.0: every question is generated by an LLM conditioned on prior answers. The architect's Path C frame: a new `turingos llm complete` subcommand, with the web layer holding per-session conversation state.

This document does NOT ask "what does Software 3.0 want". It asks: **TuringOS already has ChainTape, CAS, predicates, `PromptCapsule`, `AttemptTelemetry`, system tx, evidence capsules, sequencer admission rules, shielding policy, FC1/FC2/FC3 invariants. What is the redesign that maximally re-uses those primitives and adds the minimum new surface, while honoring every existing invariant?**

The answer falls out almost mechanically once you list the primitives.

**The strong claim**: every per-turn artifact the new grill needs already has an existing CAS object type or capsule. We do not need to invent new tape events.

### 1.2 The map from a single grill turn to existing CAS objects

One LLM-driven grill turn = "given prior Q/A pairs, generate next Q (or termination signal); user answers; predicate inspects the Q and the A; conditionally extend the conversation."

Per turn, three CAS objects should be written:

```
GrillTurnT_t:
  ├── PromptCapsule          (ObjectType::PromptCapsule)
  │     └── prompt_context_hash = sha256(visible_context_bytes)
  │     └── read_set = {prior_qa_pairs, system_template_id}
  │     └── visible_context_cid → bytes of the structured-history view
  │
  ├── AttemptTelemetry       (ObjectType::AttemptTelemetry)
  │     └── attempt_kind = ExternalizedLlmCycle (existing enum value)
  │     └── candidate_payload_cid → CID of parsed Q (the LLM-generated next question)
  │     └── prompt_context_hash matches PromptCapsule.prompt_context_hash
  │     └── (no lean_result_cid — Lean not invoked here)
  │
  └── EvidenceCapsule        (ObjectType::EvidenceCapsule, schema=turingos-spec-grill-turn-v1)
        └── encloses both CIDs + per-turn predicate verdicts + canonical
            "turn manifest" for cheap dashboard regeneration
```

No new `ObjectType` is required. `AttemptTelemetry` already carves out a per-LLM-cycle slot. `PromptCapsule` already carves out the redacted-prompt-context slot. `EvidenceCapsule` is the standard "high-volume evidence anchored by CID" wrapper Phase 7 already uses.

### 1.3 Does this need new typed_tx variants?

**No.** The architect's Phase 7 §3 caution against new typed_tx is inherited here. A grill turn is not a state-machine event that needs sequencer admission — it is **CAS-only evidence**.

Compare:
- **WorkTx**: agent submits proof prefix → sequencer admits / rejects → L4 or L4.E
- **Grill turn**: human + LLM produce a Q/A pair → CAS write, no economic effect, no Q_t mutation

A grill turn does not move any chip on the economic board. It does not consume Coin. It does not affect agent registry. It does not change scheduling. Therefore it does not earn the right to a typed_tx slot.

It is **derived signal** in the sense of Art. 0.2's "派生视图" (derived view). The conversation state is reconstructable from the per-turn capsules + the spec_capsule. The grill's terminal output (`spec.md` + final EvidenceCapsule) is what Phase 7 already anchors.

### 1.4 FC1 compliance check

FC1 hard invariant (CLAUDE.md §3.1):
```
externalized_attempt_count
=
  L4_WorkTx_attempt_count
+ L4E_WorkTx_rejection_count
+ explicitly_anchored_capsule_attempt_count
```

Each LLM-generated question and the synthesis is an **externalized attempt** in the FC1 sense — its output is used in future prompt context (FC1 explicit clause: "used in future prompt context"). Therefore each LLM call must be represented. We satisfy the invariant by adding to the third bucket: `explicitly_anchored_capsule_attempt_count` via the EvidenceCapsule wrapping each turn.

A turn's EvidenceCapsule has the form (Phase-7 already uses this pattern via `spec_capsule.rs`):
- `schema_id = "turingos-spec-grill-turn-v1"`
- contains `prompt_capsule_cid`, `attempt_telemetry_cid`, `predicate_verdicts[]`, `turn_index`, `parent_turn_cid`
- this counts as one explicitly-anchored attempt in the FC1 ledger

The web-layer counter that the Phase 7 grill currently exposes (1 LLM call for the synthesis) becomes N+1 anchored attempts (N turns + 1 synthesis). This is the only update to the FC1 accounting view that is required, and it is mechanical.

## 2. The flowchart-faithful design

### 2.1 Tape events (zero new typed_tx)

| Event | Where it lands | Object type |
|---|---|---|
| Session opens | (none — implicit; per-session workspace dir created) | filesystem-only |
| Per-turn LLM call | CAS write | `AttemptTelemetry` + `PromptCapsule` |
| Per-turn predicate verdict | embedded in EvidenceCapsule schema | `EvidenceCapsule` |
| Per-turn evidence rollup | CAS write | `EvidenceCapsule` (schema `turingos-spec-grill-turn-v1`) |
| Conversation rollup | CAS write | `EvidenceCapsule` (schema `turingos-spec-grill-session-v1`) — chains all turn CIDs |
| Final spec.md | CAS write | `EvidenceCapsule` (schema `turingos-spec-capsule-v1`, **unchanged from Phase 6.3**) |

The Phase-6.3 `turingos-spec-capsule-v1` capsule **does not change**. It still wraps `spec.md`. What we add is two new capsule **schemas**:

- `turingos-spec-grill-turn-v1`: rolls up one turn's evidence (PromptCapsule CID + AttemptTelemetry CID + verdicts + parent_turn_cid for the chain)
- `turingos-spec-grill-session-v1`: rolls up the full session (turn_cids[] + final spec capsule CID + termination_reason + session_id)

Adding capsule **schemas** is Class-2 (it does not touch `src/bottom_white/cas/schema.rs::ObjectType` — we re-use `EvidenceCapsule`). The schemas are identified by `schema_id` strings stored in CAS metadata, exactly like `SPEC_CAPSULE_SCHEMA_ID = "turingos-spec-capsule-v1"` is today.

This is the entire **constitutional surface** of the change. Everything else is glue.

### 2.2 Why not new typed_tx?

- Class-4 surface freeze (AGENTS.md §6): `typed_tx.rs` schema is restricted; a new variant requires explicit architect ratification.
- CLAUDE.md §19: "no manipulation by sequencing — do not close easy gaps to create progress optics while load-bearing blockers remain red". G-016/G-019/G-021/G-028 (PromptCapsule landing) is already an open architect concern; introducing a parallel `LlmCompletionTx` would create an *alternative* path to the same problem.
- The grill does not affect `EconomicState`, scheduler, market, or Q_t in any way that needs a Sequencer admission decision. CAS evidence is sufficient.

### 2.3 Replayability from genesis_report + ChainTape + CAS

FC2-N16 says: every real evidence run must be replayable from `genesis_report + ChainTape + CAS + agent_registry + system_pubkeys`. Our redesign satisfies this:

- `genesis_report` already pins constitution hash + system pubkeys (Phase 6.3).
- ChainTape (L4 / L4.E) has no per-turn entries — but it doesn't need them; the grill is CAS-resident evidence.
- CAS holds `PromptCapsule[]`, `AttemptTelemetry[]`, `EvidenceCapsule[]` chained by `parent_turn_cid`.
- A future `cargo replay --chaintape <ws>` invocation can walk the session-rollup capsule, fetch each turn capsule by CID, verify each AttemptTelemetry's `prompt_context_hash == PromptCapsule.prompt_context_hash`, and reproduce the conversation deterministically (modulo the LLM's nondeterminism — see §8 Risk #2).

## 3. The predicate / oracle design

The architect's pinpoint concern: "predicate boundaries on LLM output". Constitutional AI in 2025-2026 has converged on the same architecture in different vocabulary: **constrained generation + boundary predicates + retry-on-fail** (cf. structured-output literature: XGrammar, OpenAI strict mode, Anthropic native structured output Q4 2025). TuringOS already encodes this in `PredicateResultsBundle.acceptance` (boolean predicates with `BoolWithProof`).

### 3.1 The five predicates that fire per turn

For each per-turn LLM call, the following predicates run **after** the LLM responds and **before** we accept the turn:

#### P1 — `output_parses_as_grill_turn_payload`
- **Definition**: LLM output must parse as the typed turn-payload JSON schema (see §3.2). Constructed-decoded with JSON-Schema if SiliconFlow's API surface supports it; otherwise post-hoc parse + retry.
- **Fail**: write `RejectionClass::ParseFailed` analog into the turn capsule; retry up to N=3; on terminal failure, emit a turn capsule with `outcome = AttemptOutcome::ParseFail` (existing enum value!) and abort the session with a typed error to the user.
- **FC route**: equivalent to FC1 predicate-fail → CAS-anchored capsule (no L4.E required because there's no typed_tx; the EvidenceCapsule itself is the rejection evidence).

#### P2 — `output_is_question_or_terminator`
- **Definition**: the turn payload's `kind` field must be `"question"` or `"terminate"`. Anything else (assistant statement, refusal, hallucinated meta-comment) fails.
- **Fail**: same retry-up-to-N flow as P1.
- **Rationale**: this is the Software-3.0 / Voss-mirror loop discipline. The LLM is not allowed to break character.

#### P3 — `output_does_not_reference_unread_keys`
- **Definition**: the question text MUST NOT cite a fact that is not in the structured-history `read_set`. This is the anti-hallucination boundary.
- **Implementation**: this is a PCP predicate (Art. I.1.1 — "high-probability rejection, completeness=1"). Hard-detect overt cases: the question must not contain content from prior **assistant turns** that wasn't seeded by a **user turn**. Soft-detect: cluster-distance check on entities mentioned in question vs entities seen in user answers (deferred to v2).
- **Fail**: retry; on terminal fail, treat as `RejectionClass::Opaque` (per CLAUDE.md §15: do not broadcast why it failed, just record the structural mismatch).

#### P4 — `output_does_not_leak_shielded_zone`
- **Definition**: question must not include content that the per-session shielding policy (§4) marks as private. For Phase 7-N, the only shielded zone is "internal LLM-prompt scaffolding text" (e.g. the system prompt should not be echoed back; intermediate reasoning markers like `<thinking>` should be stripped before storage and absent from generated questions).
- **Fail**: hard reject (no retry — this is a Safety-class predicate per `SafetyOrCreation::Safety`).

#### P5 — `cost_budget_not_exceeded`
- **Definition**: `total_llm_calls_this_session ≤ MAX_TURNS` (default 16; configurable per-workspace).
- **Fail**: do NOT retry. Force the LLM into "terminate" mode (synthesize spec from what we have). Emit a turn capsule with `forced_terminate = true`.
- **Rationale**: economic discipline (CLAUDE.md §13: "writes/append/challenge/verify/settle require stake/escrow/bond"). The LLM is a cost-bearing resource; bounded sessions are non-negotiable.

### 3.2 The turn-payload schema (constrained output)

```json
{
  "kind": "question" | "terminate",
  "question_text": "string (required if kind=question)",
  "rationale_brief": "string ≤ 200 chars (private — never enters user-visible UI)",
  "covered_slots": ["job", "anchor", "memory", "first_run", "robustness", "scope", "success", "playback"],
  "missing_slots": ["..."],
  "termination_reason": "string (required if kind=terminate)",
  "draft_spec_outline": "string (required if kind=terminate; minimal markdown)"
}
```

This schema is content-addressable: its SHA-256 is `system_prompt_template_hash` in `PromptCapsule`. Versioned. Bumping it bumps the policy version.

The `covered_slots` / `missing_slots` fields are the LLM's self-judgment of what the user has actually answered. Predicate P3 uses these for canonical-slot tracking (§5).

### 3.3 The predicate verdict bundle stored on each turn

Each turn capsule carries:

```rust
struct TurnPredicateBundle {
    p1_parse_ok: BoolWithProof,
    p2_kind_ok: BoolWithProof,
    p3_no_unread_ref: BoolWithProof,
    p4_no_shielded_leak: BoolWithProof,
    p5_budget_ok: BoolWithProof,
    safety_class: SafetyOrCreation::Safety,  // hard-fail-closed on P1/P2/P4
}
```

This mirrors `PredicateResultsBundle` exactly. Class-3 wire-up: re-use existing `BoolWithProof` from `src/state/typed_tx.rs`.

## 4. The shielding model for per-turn LLM context

CLAUDE.md §15 forbids "global context stuffing with historical logs". The grill needs prior conversation to generate the next question — but verbatim prompt growth is the obvious anti-pattern.

### 4.1 Decision: structured history (option C)

Three options:
- **(A) Full verbatim history** — grows linearly; size-bomb on long sessions; risk: every reply concatenated into next prompt leaks shielded fields between turns.
- **(B) Summarized history** — requires an extra LLM call per turn, which doubles cost and adds determinism risk (the summarizer can drift).
- **(C) Structured history (typed turns with fields)** — each prior turn reduces to `(question_text, answer_text, covered_slots)` plus a fixed canonical-slot table the LLM updates each turn.

**Choose C.** Specifically: the prompt for turn N contains:
1. Frozen system template (`system_prompt_template_hash` pinned)
2. **CanonicalSlotTable** — a JSON array of 8 canonical slots (job / anchor / memory / first-run / robustness / scope / success / playback) with each slot in one of `{empty, partial, satisfied}` state, last-updated turn index, and a one-line user-utterance excerpt (≤ 80 chars) when satisfied
3. **Last 3 turn excerpts** — only `question` + `answer` (no `rationale_brief`, no `covered_slots` JSON, no internal LLM reasoning)
4. Instruction to emit the §3.2 schema

This bounds context size to O(constant) regardless of session length. It honors Art. III.2 (封装细节 / progressive disclosure) and Art. III.3 (屏蔽相关性 / shield laterally-correlated info).

### 4.2 The `read_set` for turn N's PromptCapsule

```rust
PromptCapsule.read_set = {
    ReadKey("session_id"),
    ReadKey(format!("turn:{}", n-3..=n-1)),  // last 3 turns only
    ReadKey("canonical_slot_table"),
    ReadKey("system_prompt_template"),
}
```

Note: turns N-4 and earlier are **not in the read_set**. The agent at turn N has no read access to those bytes. Replay later can reconstruct them via the session-rollup capsule's `turn_cids[]` — but the per-turn LLM call cannot see them. This is the Art. III.2 enforcement.

### 4.3 What is private / shielded between turns

- `AttemptTelemetry.candidate_payload_cid` of any prior turn — never enters next prompt
- The LLM's `rationale_brief` field — stored CAS-side but stripped from next-prompt context (audit-only)
- The CanonicalSlotTable's `last-updated turn index` — visible, but the underlying telemetry IDs are not

## 5. The completion criterion

The architect's question: "LLM decides when enough info — what does this look like on TuringOS?"

### 5.1 Two layers: LLM judgment + predicate verification

**Layer 1 (LLM self-judgment)**: the LLM emits `{"kind": "terminate", "termination_reason": ..., "draft_spec_outline": ...}` when it judges sufficient coverage.

**Layer 2 (predicate verification)**: a `predicate_termination_acceptable` runs on top:
- Check that ≥ 6 of the 8 canonical slots are in `{partial, satisfied}` state in the CanonicalSlotTable.
- Check that the user has had ≥ 4 conversation turns (avoid premature termination on a chatty user who answered everything in turn 1).

### 5.2 Disagreement contract

| LLM signal | Predicate | Action |
|---|---|---|
| `terminate` | accepts | proceed to synthesis (existing Phase-6.3 LLM synthesis step) |
| `terminate` | rejects (slots empty) | inject a forced follow-up — system emits a synthetic prompt to LLM: "you signaled terminate but slots X, Y are empty. Ask the user about X first." Treat this as a Voss-mirror loop (Art. III + the existing grill methodology). |
| `question` | (predicate doesn't run) | continue |
| LLM hits budget cap (P5) | (forced) | system synthesizes a `terminate` with `forced_terminate=true` flag. The Phase-6.3 synthesis still produces a `spec.md` but the final EvidenceCapsule includes `partial_session=true` metadata. |

### 5.3 Hard upper bound

`MAX_TURNS = 16` (configurable). Soft suggestion to the LLM at turn 12+: "you have 4 turns left; prioritize unfilled slots". This is in the system template, not a runtime mutation of the prompt.

The hard bound preserves replayability under cost-attack scenarios. A malicious user cannot force the LLM into infinite generation by giving deliberately uninformative answers.

## 6. Replay + audit story

### 6.1 The auditor's view

A future `cargo replay --chaintape <ws> --session <session_id>` invocation:

1. Open the session-rollup `EvidenceCapsule` (schema `turingos-spec-grill-session-v1`) by CID. It contains `turn_cids[]`, `final_spec_capsule_cid`, `session_id`.
2. Walk `turn_cids[]` in order. For each turn:
   - Open the turn capsule (schema `turingos-spec-grill-turn-v1`)
   - Resolve `prompt_capsule_cid` → fetch PromptCapsule; verify `hidden_fields_redacted == true`
   - Resolve `attempt_telemetry_cid` → fetch AttemptTelemetry
   - Verify `AttemptTelemetry.prompt_context_hash == PromptCapsule.prompt_context_hash` (Phase-6.3 invariant already in `prompt_capsule_referenced_by_attempt_telemetry` gate)
   - Resolve `candidate_payload_cid` → fetch parsed turn-payload JSON; verify it parses against the v1 schema
   - Replay all 5 predicates against the parsed payload + canonical slot table at that turn; verdict must match the stored `predicate_verdicts`
3. After last turn, fetch `final_spec_capsule_cid` → verify it's the existing `turingos-spec-capsule-v1` capsule
4. Verify the synthesis input — the spec capsule's body must be derivable from the conversation history (this is **not** byte-exact reproducible because of LLM nondeterminism; see Risk §8.2 for mitigation)

### 6.2 What the auditor sees if a turn was rejected

A rejected turn (predicate-fail) still has a capsule:
```
EvidenceCapsule (schema=turingos-spec-grill-turn-v1)
  ├── prompt_capsule_cid: <set>
  ├── attempt_telemetry_cid: <set, outcome=ParseFail>
  ├── predicate_verdicts: { p1_parse_ok = false, … }
  ├── retry_index: 0..2
  └── final_outcome: "retried" | "abort_session"
```

Each retry emits its own AttemptTelemetry + PromptCapsule (because the prompt to the LLM differs on retry — usually appending "your previous response failed predicate X"). So the FC1 ledger sees N AttemptTelemetry rows for N retries, plus 1 final accepted AttemptTelemetry, all anchored from the same turn capsule.

### 6.3 Reproducibility test

A constitution gate (Class 2 test addition) should assert:
```rust
fn grill_session_replay_yields_identical_conversation_view() {
    let session_capsule = read_session(...);
    let reconstructed = derive_conversation_view_from_cas(&session_capsule);
    let dashboard_view = read_dashboard_session(...);
    assert_eq!(reconstructed, dashboard_view);
}
```

This is the FC2 "all signals reconstructable from tape" gate, applied to the grill.

## 8. Risks to TuringOS constitutional integrity

### Risk 1 — Prompt leakage between sessions if AppState not isolated
- **Vector**: `AppState.sessions: HashMap<SessionId, ConversationState>` lives in-memory. A buggy session-key collision could let session B see session A's prior turns when assembling the next prompt.
- **Mitigation**: session IDs already validated via `is_safe_session_id` (§web/spec.rs:424). Add a Class-1 test that asserts `read_set` for any per-turn PromptCapsule contains only keys prefixed with that turn's `session_id`. Make ConversationState a `BTreeMap<SessionId, …>` and add an explicit `assert!(!self.sessions.contains_key(new_id))` on session creation.
- **Constitution anchor**: Art. III.3 (屏蔽相关性 — shield lateral correlation between agents/sessions).

### Risk 2 — LLM nondeterminism breaks deterministic replay
- **Vector**: SiliconFlow chat-completion is nondeterministic (sampling temperature > 0; even temp=0 doesn't guarantee bit-exact across days). FC2 says "every real evidence run must be replayable from genesis_report + ChainTape + CAS". If we re-call the LLM at audit time, we get different output → replay diverges.
- **Mitigation**: replay does **not** re-invoke the LLM. The verbatim turn payload at audit time is fetched from `candidate_payload_cid` (CAS-resident). The replay reconstructs the **conversation view** (Q's and A's the user saw), not "what would the LLM say today". This is consistent with how `LeanResult` works: the proof verdict is canonical evidence, not a re-run of Lean.
- **Architectural cost**: we must store enough of the LLM's output to make the conversation reconstructable. That's exactly what `candidate_payload_cid` does. The privacy fence (Art. III.1, CLAUDE.md §15) requires that we do NOT store the raw LLM response (including hidden CoT / chain-of-thought). Per `attempt_telemetry.rs:38-53` privacy invariant: candidate_payload is **parsed** bytes, not raw. So we store the parsed turn-payload JSON, not the model's verbatim API response.
- **Constitution anchor**: Art. 0.2 Tape Canonical 公理 (all signals reconstructable from tape) + CR-18R.4 v2 (no raw response on tape).

### Risk 3 — Per-turn LLM latency creates user-facing UX gap
- **Vector**: each turn requires a SiliconFlow round-trip (200ms - 4s typical for DeepSeek-V3.2). Users typing fast on the frontend will see waits between answers. This is a UX risk, not a constitutional one, but it could push someone to optimize by caching LLM outputs → cache becomes shadow ledger → Art. 0.2 violation.
- **Mitigation**: (a) accept the latency as inherent; (b) optionally pre-warm the LLM after answer N-1 is submitted so the next question is ready by the time the user finishes; (c) NEVER store warmup outputs except as proper per-turn capsules — pre-warm is a UX latency strategy, not a caching mechanism.
- **Constitution anchor**: Art. 0.2 + CLAUDE.md §16 (no shadow ledger as source of truth).

### Risk 4 — Predicate P3 (no-unread-reference) is PCP-only, not perfect
- **Vector**: P3 is the anti-hallucination predicate. A skilled adversarial LLM could craft a question that *technically* doesn't cite an unread fact but is semantically out of bounds (e.g. mention an "obvious common-knowledge" fact that wasn't in user input).
- **Mitigation**: Art. I.1.1 explicitly endorses PCP predicates: "completeness=1, soundness error 极小". So P3 is a PCP predicate by design. We pair it with two reinforcing predicates: P2 (kind discriminator) and P4 (no shielded zone leak). Stack of three PCPs > one perfect.
- **Constitution anchor**: Art. I.1.1 (PCP 谓词与"疑罪从无").

### Risk 5 — Cost-budget gaming
- **Vector**: a malicious user types 4096-char garbage answers to inflate prompt size and burn through MAX_TURNS quickly without producing useful spec.
- **Mitigation**: P5 (cost budget) is per-session in **turn count**, not in token count, so 4096-char answers do not gain extra turns. The structured-history shielding (§4) bounds context size regardless of answer length. The frontend already validates 4096-char ceiling per answer (validate_answers in spec.rs:391-417).
- **Constitution anchor**: Law 2 (Only Investment Costs Money) — the user pays no Coin here but the system bears LLM cost; structural defense (turn cap) is the right abstraction.

### Risk 6 (additional) — Shielding gap on `rationale_brief` field
- **Vector**: the §3.2 turn-payload schema includes `rationale_brief` (≤ 200 chars LLM self-explanation). If this leaks into the next turn's prompt or the user-visible UI, it could (a) leak LLM private CoT (Art. III private-CoT discipline), or (b) make the LLM "see its own thinking" across turns and entrain on it (Art. III.3 lateral correlation).
- **Mitigation**: §4.1 already excludes `rationale_brief` from the structured-history view. Add a Class-1 test `rationale_brief_never_appears_in_subsequent_prompt_capsule_visible_context`. The field exists for audit-only.
- **Constitution anchor**: Art. III.1 + CLAUDE.md §15 (forbidden: raw autopsy broadcast / global context stuffing).

### The key claim

The architect's Path C frame is correct, but it can be implemented **without inventing new constitutional primitives**. The existing `PromptCapsule` + `AttemptTelemetry` + `EvidenceCapsule` triad already carves out exactly the slot a per-turn LLM cycle needs. The new surface is:
- 2 new `schema_id` strings (Class 2-3, identical pattern to Phase 6.3's `turingos-spec-capsule-v1`)
- 1 new CLI subcommand action (`llm complete`, Class 2)
- 1 new web endpoint (`POST /api/spec/turn`, Class 2)
- ~750 LOC of glue across 5 files

No Class 4. No sequencer admission change. No typed_tx schema mutation. No new ObjectType. The substrate is sufficient as-is.
